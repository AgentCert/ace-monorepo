# 1.1 — Why AgentCert

## The Problem We Were Trying to Solve

We were building AI agents to manage Kubernetes infrastructure — detecting faults, diagnosing root causes, running remediation. The agent worked well in our demos. It could spot a crashed pod, reason through the logs, and recover the service. Impressive, honestly.

But when it came time to hand it over to a customer and say "this is production-ready," we hit a wall. We had no credible way to answer the questions that actually matter:

- How often does the agent catch a fault? Not in one demo run — across 30 independent runs?
- When it detects a fault, how fast is it? What's the worst-case response time, not just the average?
- Does it perform the same on network faults as it does on pod crashes? Or does it quietly fall apart on specific fault types?
- Can we prove, with statistical confidence, that it meets an SLA?
- Is it hallucinating? Is it exposing sensitive data? Is it safe to run?

Unit tests couldn't answer these. Integration tests couldn't either. And a human manually reviewing agent runs at scale wasn't realistic. We needed something different.

---

## Why Traditional Testing Isn't Enough

The fundamental problem with AI agents is that they are **non-deterministic by design**. The same fault, the same environment, the same prompt — and the agent might take a different path every single time. Sometimes it's faster. Sometimes it hallucinates. Sometimes it misses the root cause entirely.

This means a single test run tells you almost nothing. You got lucky or unlucky. To say anything meaningful about an agent's reliability, you need to run it dozens of times, under controlled conditions, and reason statistically about what you observe.

Traditional testing frameworks aren't built for this. They assume deterministic outputs. Pass or fail. Same input, same output. None of that holds for LLM agents.

On top of that, the kinds of failures we care about are **operational failures** — the agent doesn't detect a pod deletion fast enough, it misdiagnoses a network fault as a resource issue, it escalates unnecessarily and takes down a healthy service. These failures only emerge when you inject real chaos into a real environment and watch what happens. You can't unit test your way to that answer.

---

## What We Built Instead

AgentCert is a **fault injection + statistical certification pipeline**. The idea is straightforward, even if the implementation has depth:

1. Inject controlled infrastructure faults into a Kubernetes cluster using LitmusChaos
2. Let the AI agent under test respond to those faults — completely on its own, with no hints
3. Capture everything the agent does via Langfuse traces
4. Run those traces through a measurement pipeline that extracts quantitative and qualitative metrics
5. Repeat 30 times per fault type to get statistically meaningful results
6. Aggregate, test hypotheses, and produce a certification report that tells you exactly what you can and cannot claim about this agent

The output is not a pass/fail test result. It is a **certification report** — the same kind of structured evidence you would want before deploying any safety-critical system. It tells you the detection rate with confidence intervals. It tells you where the agent is consistent and where it is erratic. It surfaces the hallucination patterns. It gives you concrete recommendations for what to fix before you deploy.

## The Questions AgentCert Is Designed to Answer

These are the questions a principal engineer or a customer should be able to ask and get a data-backed answer to:

| Question | What AgentCert Provides |
|---|---|
| How reliably does this agent detect faults? | Detection rate with Wilson confidence interval and certified lower-bound floor |
| How fast does it respond? | Time-to-detect and time-to-mitigate: median, P95, bootstrap CI across 30 runs |
| Does it meet our SLA? | Wilcoxon one-sample test against the SLA threshold — pass, conditional, or fail |
| Is it consistent, or all over the place? | Coefficient of variation and Levene's test per fault category |
| Does it handle network faults as well as pod crashes? | Cross-category comparison with Kruskal-Wallis and effect sizes |
| Is it hallucinating? | Hallucination score per run, aggregated with combined judge across all runs |
| Is it safe? | RAI compliance rate, PII detection, adversarial prompt handling — worst-case propagation |
| What should we fix first? | Evidence-based recommendations grounded in the actual experimental findings |

---

## One More Thing Worth Saying

AgentCert is not a silver bullet. It tells you how the agent performs under the specific fault conditions you tested, in the specific environment you set up, with the specific LLM version that was running at the time. Change the model, change the environment, change the fault configuration — and you should re-certify.

Think of it less like a unit test suite and more like a **performance audit**. It gives you an honest, reproducible snapshot of what this agent is capable of today. That snapshot is enormously useful — but it is a snapshot, not a guarantee.

That is the honest framing. Everything else in this wiki is about how we built the machinery to produce that snapshot reliably.
