# 1.2 — High-Level Workflow

## The Short Version

You inject a fault into a Kubernetes cluster. An AI agent tries to detect and fix it. You capture everything the agent does. You repeat this 30 times. A pipeline measures what actually happened across all those runs and produces a certification report.

That is the entire thing. Everything else is implementation detail.

---

## The Full Picture

Here is the end-to-end flow, from the moment you decide to test a fault to the moment you have a report in your hands.

## Who Does What

It is worth being explicit about which parts are automated and which require human input.

| Step | Who Does It | Human Input Required? |
|---|---|---|
| Define experiment | You (the experiment designer) | Yes — fault configuration and ground truth must be authored |
| Inject fault | LitmusChaos (automated) | Once, to configure the chaos experiment |
| Agent responds | The AI agent under test (automated) | No — agent operates autonomously |
| Capture trace | Langfuse (automated) | No |
| Fault bucketing | AgentCert Phase 0 (automated) | No |
| Metrics extraction | AgentCert Phase 1 (automated) | No |
| Aggregation | AgentCert Phase 2 (automated) | No |
| Hypothesis testing (optional, n ≥ 30) | AgentCert Phase E (automated) | No |
| Certification report | AgentCert Phase 3 (automated) | No |
| Review the report | You (the certifier consumer) | Yes — a human should always review before acting on results |

The entire measurement pipeline runs without human involvement once the experiment is set up. The only things that require a human are writing the fault configuration upfront, and reading the certification report at the end.

## What a Typical Experiment Looks Like

To make this concrete — here is what running a pod-delete certification experiment looks like in practice:

1. Write the fault configuration for `pod-delete`: specify the target namespace, the injection timing, the ideal course of action, the ideal tool trajectory
2. Configure LitmusChaos to inject a pod deletion every ~5 minutes
3. Start the agent and let it run autonomously against the live cluster
4. Collect 30 traces — this typically takes a few hours depending on chaos timing and agent response speed
5. Run the certifier pipeline against all 30 traces
6. Review the certification report

At the end you have a report that tells you, with statistical backing: this agent detects pod deletions reliably (or it doesn't), it responds within a certain time window (or it doesn't), and here is exactly where it falls short and what to do about it.

---

## The Key Thing to Understand

The certifier does not evaluate a single run. It evaluates a **distribution of runs**. The insight is that a single run tells you very little about a non-deterministic system. What matters is the pattern across many runs — the central tendency, the variance, the tail behaviour, the safety record.

Everything in this pipeline is designed around that insight. Thirty runs per fault. Statistical aggregation. Confidence intervals on every key metric. Worst-case propagation for safety. That is why this works.
