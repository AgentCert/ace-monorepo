# 1.2 — High-Level Workflow

## The Short Version

You inject a fault into a Kubernetes cluster. An AI agent tries to detect and fix it. You capture everything the agent does. You repeat this 30 times. A pipeline measures what actually happened across all those runs and produces a certification report.

That is the entire thing. Everything else is implementation detail.

---

## The Full Picture

Here is the end-to-end flow, from the moment you decide to test a fault to the moment you have a report in your hands.

```
┌─────────────────────────────────────────────────────────────────────────┐
│  STEP 1 — DEFINE THE EXPERIMENT                                         │
│                                                                         │
│  Before any fault is injected, you write a fault configuration file.   │
│  This is the ground truth — it says exactly what fault you are         │
│  injecting, what the agent should ideally do, and what tools it        │
│  should call in what order.                                             │
│                                                                         │
│  The agent never sees this file. It is purely for evaluation.          │
│  Think of it as the answer key that gets compared against the          │
│  agent's actual answers after the fact.                                │
│                                                                         │
│  → Output: fault_configuration.json (kept secret from the agent)       │
└────────────────────────────┬────────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  STEP 2 — INJECT THE FAULT  (LitmusChaos)                               │
│                                                                         │
│  LitmusChaos injects a controlled fault into the Kubernetes cluster.   │
│  This could be a pod deletion, a network packet loss, a CPU hog,       │
│  a disk fill — any of the fault types in the taxonomy.                 │
│                                                                         │
│  The fault runs for a defined duration (typically 30–120 seconds)      │
│  and then the cluster self-heals. The agent is not told anything       │
│  about the fault. It has to figure it out from what it observes.       │
│                                                                         │
│  → Output: fault is live in the cluster; injection timestamp recorded  │
└────────────────────────────┬────────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  STEP 3 — AGENT RESPONDS  (The AI Agent Under Test)                     │
│                                                                         │
│  The AI agent monitors the cluster via MCP tools — kubectl,            │
│  log queries, health checks, metrics. It detects the anomaly,          │
│  diagnoses the root cause, and attempts remediation.                   │
│                                                                         │
│  Everything the agent does is automatically captured in Langfuse:      │
│  every LLM call, every tool invocation, every reasoning step,          │
│  every output. This becomes the raw trace for that run.                │
│                                                                         │
│  → Output: Langfuse trace JSON (one file per run)                      │
└────────────────────────────┬────────────────────────────────────────────┘
                             │
                             │  Repeat steps 2–3 approximately 30 times
                             │  for each fault type
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  STEP 4 — FAULT BUCKETING  (Phase 0 of the Certifier)                  │
│                                                                         │
│  A single trace file may contain events from multiple overlapping      │
│  faults. The first thing the certifier does is separate them —         │
│  assigning each trace event to the fault it belongs to.                │
│                                                                         │
│  Deterministic where possible (events that clearly belong to one       │
│  fault), LLM-assisted where there is ambiguity (events during          │
│  periods when multiple faults were active simultaneously).             │
│                                                                         │
│  → Output: one JSON bucket per fault, per run                          │
└────────────────────────────┬────────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  STEP 5 — METRICS EXTRACTION  (Phase 1 of the Certifier)               │
│                                                                         │
│  For each fault bucket, the certifier extracts the metrics that        │
│  matter: how long did detection take, how long did mitigation take,    │
│  how good was the reasoning, did the agent hallucinate, was it safe.   │
│                                                                         │
│  An LLM reads the trace and pulls out observations. Python code        │
│  handles all the arithmetic — the LLM is never asked to do math.      │
│                                                                         │
│  → Output: one ExtractionResult document per run, stored in MongoDB    │
└────────────────────────────┬────────────────────────────────────────────┘
                             │
                             │  After all ~30 runs are extracted
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  STEP 6 — AGGREGATION  (Phase 2 of the Certifier)                      │
│                                                                         │
│  Now we have ~30 measurements per fault type. The certifier            │
│  aggregates them:                                                       │
│                                                                         │
│   Numbers → statistical aggregation in Python code                     │
│              (mean, median, P95, confidence intervals)                 │
│                                                                         │
│   Narratives → LLM Council                                             │
│              (multiple independent judges read all 30 run summaries    │
│               and produce a consensus assessment)                      │
│                                                                         │
│   Safety → worst-case propagation                                      │
│              (one failure in any run = category flagged)               │
│                                                                         │
│  → Output: certification scorecard per fault category                  │
└────────────────────────────┬────────────────────────────────────────────┘
                             │
                             ▼  (optional, when n ≥ 30 per fault category)
┌─────────────────────────────────────────────────────────────────────────┐
│  STEP 7 — STATISTICAL HYPOTHESIS TESTING  (Phase E)                    │
│                                                                         │
│  When you have enough data, the pipeline runs 9 formal statistical     │
│  tests. These go beyond summary statistics — they give you confidence  │
│  intervals, prove SLA compliance, detect cross-category weaknesses,    │
│  and check whether the agent's performance is drifting over time.      │
│                                                                         │
│  The results are packaged as Appendix A1–A5 and passed directly        │
│  into Step 8, where they appear at the end of the report.              │
│                                                                         │
│  → Output: hypothesis test results (Appendix A1–A5)                    │
└────────────────────────────┬────────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  STEP 8 — CERTIFICATION REPORT  (Phase 3 of the Certifier)             │
│                                                                         │
│  The scorecard — plus the hypothesis appendix if it was computed —     │
│  is turned into a structured 12-section report. Deterministic          │
│  builders create the tables, charts, and scoring cards. LLM narrative  │
│  builders — running in parallel — write the prose.                     │
│                                                                         │
│  The output is validated against a strict schema. If anything is       │
│  malformed, the pipeline errors rather than producing a bad report.    │
│                                                                         │
│  → Output: certification_report.json → HTML + PDF                      │
└─────────────────────────────────────────────────────────────────────────┘
```

## Who Does What

It is worth being explicit about which parts are automated and which require human input.

| Step | Who Does It | Human Input Required? |
|---|---|---|
| Define experiment | You (the experiment designer) | Yes — fault configuration and ground truth must be authored |
| Inject fault | LitmusChaos (automated) | Once, to configure the chaos experiment |
| Agent responds | The AI agent under test (automated) | No — agent operates autonomously |
| Capture trace | Langfuse (automated) | No |
| Fault bucketing | AgentCert Phase 0 (automated) | No |
| Metrics extraction | AgentCert Phase 1 (automated) | No |
| Aggregation | AgentCert Phase 2 (automated) | No |
| Hypothesis testing (optional, n ≥ 30) | AgentCert Phase E (automated) | No |
| Certification report | AgentCert Phase 3 (automated) | No |
| Review the report | You (the certifier consumer) | Yes — a human should always review before acting on results |

The entire measurement pipeline runs without human involvement once the experiment is set up. The only things that require a human are writing the fault configuration upfront, and reading the certification report at the end.

---

## What a Typical Experiment Looks Like

To make this concrete — here is what running a pod-delete certification experiment looks like in practice:

1. Write the fault configuration for `pod-delete`: specify the target namespace, the injection timing, the ideal course of action, the ideal tool trajectory
2. Configure LitmusChaos to inject a pod deletion every ~5 minutes
3. Start the agent and let it run autonomously against the live cluster
4. Collect 30 traces — this typically takes a few hours depending on chaos timing and agent response speed
5. Run the certifier pipeline against all 30 traces
6. Review the certification report

At the end you have a report that tells you, with statistical backing: this agent detects pod deletions reliably (or it doesn't), it responds within a certain time window (or it doesn't), and here is exactly where it falls short and what to do about it.

---

## The Key Thing to Understand

The certifier does not evaluate a single run. It evaluates a **distribution of runs**. The insight is that a single run tells you very little about a non-deterministic system. What matters is the pattern across many runs — the central tendency, the variance, the tail behaviour, the safety record.

Everything in this pipeline is designed around that insight. Thirty runs per fault. Statistical aggregation. Confidence intervals on every key metric. Worst-case propagation for safety. That is why this works.
