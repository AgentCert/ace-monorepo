# 1.3 — Certifier Assumptions

## What This Section Covers

These are the assumptions baked into the AgentCert pipeline itself — design decisions that affect how the certifier collects, interprets, and reports results. They apply regardless of which agent you are certifying or which experiment you are running.

For agent-specific and experiment-specific assumptions — SLA thresholds, environment setup, and onboarding checklist — see `2.0-Experiment-Assumptions.md` in the Experiment Design section.

---

## Reference Setup

This certification was validated against a specific configuration. Understanding what we used helps clarify where the assumptions come from — and what might need to change if you are onboarding a different agent.

**Agent under test — Flash Agent**
We used [Flash Agent](https://github.com/AgentCert/flash-agent), an extension of the AIops Lab's Kubernetes operations tooling. Flash Agent monitors clusters, detects infrastructure anomalies, and attempts autonomous remediation via MCP tool calls. It was built specifically for this use case and is the primary agent this certification evidence covers.

The architecture was designed around Flash Agent. If you are onboarding a different agent, some components — particularly trace parsing and tool trajectory matching — may need tweaking to match how that agent structures its tool calls and reasoning steps.

**Fault injection — chaos charts**
LitmusChaos experiment definitions (pod-delete, network-loss, cpu-hog, disk-fill) are published at [github.com/AgentCert/chaos-charts](https://github.com/AgentCert/chaos-charts). These are pre-configured with the metadata structure the certifier's bucketing algorithm expects.

**Trace dataset**
All Langfuse traces collected during the certification experiments are at [github.com/AgentCert/AgentCert-Traces](https://github.com/AgentCert/AgentCert-Traces). This is the canonical data source for the results in this wiki.

## Infrastructure Assumptions

### Langfuse is the trace store

The certifier consumes Langfuse trace JSON files. Every LLM call the agent makes, every tool invocation, every reasoning step — Langfuse captures this. The certifier reads timestamps, tool names, and LLM outputs directly from this trace structure.

If your agent doesn't trace to Langfuse, you need to produce a trace format that matches the structure the certifier expects. That is possible but requires engineering work on the trace reader.

### LitmusChaos is the fault injector

The bucketing algorithm (Phase 0) is designed around the timing and metadata that LitmusChaos produces — specifically, per-experiment start/end timestamps and a structured chaos engine name. The deterministic bucketing pass relies on these to assign trace events to the correct fault window.

If you replace LitmusChaos with a different fault injector, the metadata extraction step will need adaptation and LLM-assisted disambiguation may behave differently.

### Fault configuration includes ground truth and ideal trajectory

Each fault experiment must have a corresponding fault configuration file. This is what the certifier evaluates against. It specifies:

- What fault is being injected (type, target, parameters)
- The ideal course of action the agent should take
- The ideal tool trajectory — which tools, in what order
- SLA thresholds — the maximum acceptable time-to-detect and time-to-mitigate for this fault type

Without a fault configuration, the pipeline can extract raw metrics but cannot score against expected behaviour or evaluate SLA adherence.

### Ground truth is never shown to the agent

This is a hard requirement. The fault configuration file is used only during evaluation. If the agent can see it, you are not testing the agent — you are testing whether it can read an answer key. The pipeline cannot enforce this at runtime; it must be enforced operationally.

### SLA thresholds are defined per fault type

The certifier evaluates SLA compliance based on thresholds you define in the fault configuration. A pod deletion SLA might be 30 seconds TTD; a network degradation SLA might be 90 seconds. The certifier does not impose universal thresholds — it evaluates the agent against the thresholds that are appropriate for each fault type. If SLA thresholds are not defined, Phase E SLA compliance tests will not run.

## Design Principles

These are not configurable — they are the architecture.

### LLMs never do arithmetic

The certifier uses LLMs for three things: classifying trace events (Phase 0), extracting qualitative observations (Phase 1), and synthesising narratives across runs (Phase 2). It never asks an LLM to compute a mean, compare a number against a threshold, or evaluate SLA compliance.

All arithmetic is pure Python. Every mean, every P95, every confidence interval, every hypothesis test is computed in code. If you see a numeric claim in the certification report, it was computed by code, not narrated by a model.

### Safety uses worst-case propagation

The safety and RAI section does not aggregate failures — it propagates them. One hallucination involving sensitive data across 30 runs means the hallucination category is flagged. One adversarial prompt handling failure means that category is flagged.

This is intentional. For performance metrics, an occasional bad run is a tail-risk concern. For safety, an occasional bad run is a categorical failure. A 97% safety compliance rate is not acceptable.

### LLM Council reduces single-model bias

Phase 2 uses a council of k independent judges (typically 3–5) plus a meta-judge that synthesises a consensus. No single model's biases dominate the narrative assessment. The meta-judge reconciles diverging reads — not by picking a winner, but by producing an honest assessment that accounts for what multiple independent evaluators saw.
