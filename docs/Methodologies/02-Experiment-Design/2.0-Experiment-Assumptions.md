# 1.4 — Experiment Assumptions

## What This Page Covers

These are the assumptions specific to the Flash Agent certification experiment — the reference run that this wiki documents. They describe the agent we tested, the data we collected, the environment we ran in, and what you need to know if you want to onboard a different agent.

---

## The Reference Setup

### Flash Agent

The agent we certified is **Flash Agent** — an AIops Lab project that monitors Kubernetes clusters, detects infrastructure anomalies, and attempts autonomous remediation. It was built specifically for this kind of operational use case and is the primary agent this certification evidence covers.

Repository: [github.com/AgentCert/flash-agent](https://github.com/AgentCert/flash-agent)

Flash Agent is an extension of the AIops Lab's existing Kubernetes operations tooling. It calls the cluster via MCP tools — kubectl, log queries, health checks, metrics endpoints — and uses a reasoning LLM to decide what to do next.

### Chaos Charts

Fault injection was performed using a set of custom LitmusChaos experiment definitions covering pod-delete, network-loss, CPU-hog, and disk-fill.

Repository: [github.com/AgentCert/chaos-charts](https://github.com/AgentCert/chaos-charts)

These charts are pre-configured with the timing and metadata structure the certifier's bucketing algorithm expects. If you are using different LitmusChaos charts, verify they produce the same metadata fields.

### Trace Dataset

All Langfuse traces collected during the certification experiments are published at:

Repository: [github.com/AgentCert/AgentCert-Traces](https://github.com/AgentCert/AgentCert-Traces)

This is the canonical data source for the results reported in this wiki. If you want to reproduce the findings, re-run the certifier against these traces.

## Environment Assumptions

### The Kubernetes cluster is isolated

Flash Agent was tested against a dedicated Kubernetes cluster — not shared with production workloads. This matters because:

- Background incidents from unrelated workloads would contaminate the detection metrics
- Fault injection requires cluster-admin permissions that should not exist in production
- Reproducibility depends on a known baseline state before each fault injection

If your environment has background noise from unrelated incidents, the certifier will still run — but the detection rates and response times will reflect real anomalies mixed in with the injected ones. That affects interpretation.

### Flash Agent has full MCP tool access

During the experiments, Flash Agent had access to the full set of MCP tools it uses in production: kubectl, log queries, health checks, and metrics endpoints. We did not restrict tool access to simulate degraded conditions.

If you are certifying an agent with a different tool set, update the ideal tool trajectory in the fault configuration accordingly. The certifier scores tool use against the trajectory you define — not against Flash Agent's specific tools.

## SLA Thresholds Used

The following SLA thresholds were used for the Flash Agent certification experiments. These are documented here as the reference baseline; they are not hardcoded in the certifier and should be adjusted to match your operational requirements.

| Fault Type | TTD SLA | TTM SLA |
|---|---|---|
| pod-delete | 30 seconds | 120 seconds |
| network-loss | 60 seconds | 180 seconds |
| cpu-hog | 45 seconds | 150 seconds |
| disk-fill | 60 seconds | 180 seconds |

TTD = time-to-detect. TTM = time-to-mitigate. Both are measured from fault injection start to the agent's first relevant action.

## Onboarding a New Agent

The certifier pipeline is agent-agnostic. Flash Agent is the reference implementation, not a hard dependency. If you are onboarding a different agent, here is what needs to be verified or updated:

| Area | What to Check |
|---|---|
| Trace format | Does the agent emit Langfuse-compatible traces? If not, adapt the trace reader in Phase 0. |
| Tool naming | Does the agent call tools with names that can be matched against the ideal trajectory in the fault config? |
| MCP protocol | Verify the tool invocation format is consistent — the certifier parses tool calls from trace spans. |
| Fault configuration | Rewrite ground truth, ideal trajectory, and SLA thresholds for the new agent's expected behaviour. |
| Architecture | Review the [Architecture diagram](https://microsoft.sharepoint.com/:u:/r/teams/ExternalInfosys-ISDMACC/_layouts/15/Doc.aspx?sourcedoc=%7B8ad41b7b-a2f3-4749-a1a9-f6cc04d1987a%7D&action=default&share=cQp7G9SK86JJR6Gp9swE0Zh6EgUC4GRCcdg3OVmI08UAmOqXmQ) — some components may need tweaking depending on how the new agent is structured. |

The pipeline code does not change for a new agent. What changes is the configuration and the fault configuration ground truth files.
