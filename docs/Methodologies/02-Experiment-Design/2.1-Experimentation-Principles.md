# 2.1 — Experimentation Principles

## What This Page Covers

Before you run a single experiment, you need to understand what makes an experiment valid. This page explains the core principles that govern how AgentCert experiments are designed and executed — and why violating any of them undermines the certification results.

---

## Why 30 Runs

AI agents are non-deterministic. The same fault, the same cluster state, the same agent — and you'll get a different result every time. A single run might be your best run or your worst run. You have no way to know which.

Thirty runs per fault category is the threshold where the distribution starts to stabilise. Below 30:
- Confidence intervals are too wide to make reliable claims
- The statistical hypothesis framework (Phase E) won't activate
- Outlier runs have a disproportionate effect on aggregated metrics

Above 30, returns diminish quickly. The central tendency, variance, and tail behaviour are all well-characterised by 30. You can run more if you want tighter confidence intervals — but 30 is the minimum for the certification to carry weight.

**This is not a rule we made up.** It's the sample size threshold where the bootstrap BCa confidence intervals and Wilson score floors become meaningfully narrow. Below it, the numbers are technically correct but practically untrustworthy.

---

## What Makes a Run Valid

Not every trace that Langfuse collects is a valid run for certification purposes. A run is valid if and only if:

| Criterion | Why It Matters |
|---|---|
| The fault was actively injected during the run | The agent needs something real to respond to |
| The agent operated autonomously — no human intervention | You are certifying the agent, not a human-agent team |
| The trace is complete — fault config, tool calls, and LLM generations are all present | Incomplete traces cannot be bucketed or scored |
| The agent had full access to its MCP tools | Restricted tool access changes what the agent can do and should not be mixed with full-access runs |
| The cluster was at baseline state at the start of the run | Carryover state from a previous fault contaminates detection timing |

Runs that fail these criteria should be discarded and replaced. The certifier will process whatever traces you give it — it doesn't know if a run was invalid. That judgement is yours to make before running the pipeline.

---

## Independence of Runs

Each run must be independent. That means:

- **The cluster returns to a known baseline state between runs.** If a previous fault left residue — a pod in a degraded state, an event log full of noise — the next run starts from a contaminated environment.
- **The agent's memory is reset between runs** (if the agent maintains conversational memory). A run where the agent "remembers" the previous fault is not an independent observation.
- **The fault configuration is identical across all runs for the same fault category.** You are measuring variance in the agent's response, not variance in the fault itself.

Independence is what makes the 30 runs a sample rather than a correlated sequence. If runs are not independent, the confidence intervals are incorrect — they assume independence.

---

## Ground Truth Isolation

The fault configuration — which specifies what was injected, the ideal course of action, and the ideal tool trajectory — must never be accessible to the agent under test.

This is not just a principle. If it's violated, the entire certification is invalid. There is no way for the pipeline to detect this at runtime. You have to enforce it operationally.

The ground truth file flows into Langfuse via the OTEL Collector so it can be read by the certifier during evaluation. It does not flow into the agent's context. The agent learns about the fault only from what it observes in the cluster.

---

## Controlled Environment

AgentCert is not a field study. It is a controlled experiment. The cluster you run it on should be:

- **Dedicated** — not shared with production workloads or other test suites running concurrently
- **Stable** — baseline resource usage should be predictable and low before each fault injection
- **Instrumented** — Langfuse, OTEL Collector, and MCP tools must all be functioning correctly before you start

If background incidents are happening during your experiment, the agent will respond to them. Those responses will end up in the trace. The certifier will try to separate them from the injected fault — but this adds noise and reduces the quality of the results.

---

## Repeatability

AgentCert results are only meaningful if the experiment is repeatable. That means:

- The fault configuration is stored and version-controlled — not constructed ad hoc
- The chaos charts are pinned to a specific version
- The agent version and model version are recorded at the time of the experiment
- The cluster configuration (Kubernetes version, node types, deployed workloads) is documented

If you re-run the certification six months later with a different model version or different cluster configuration, you should expect different results. That is not a bug — it is why re-certification matters. But the results of both runs should be interpretable in isolation, which requires that the setup was documented.

---

## When to Discard a Run

Discard a run if:
- The Langfuse trace is incomplete (missing fault config, tool calls, or LLM generations)
- The agent crashed or was restarted mid-run
- The cluster had an unrelated incident during the run that the agent spent time investigating
- Human intervention occurred during the run

Do not discard a run simply because the agent performed poorly. A run where the agent missed the fault entirely is a valid data point — it contributes to the detection rate. Only discard runs where the experimental conditions themselves were compromised.
