# 2.2 — Global Methods and Standards

## What This Page Covers

These are the statistical and measurement principles that apply across all metrics in AgentCert. They govern how numbers are computed and reported, regardless of which metric you are looking at. For the actual metric definitions, rubrics, and scoring formulas, see `03-Metrics/3.0-Metrics-Reference.md`.

---

## Core Principle: LLMs Identify, Code Computes

The LLM's job is to read traces and identify things — timestamps, claim types, quality dimensions. The code's job is to do arithmetic with those identified values.

No LLM is ever asked to compute a mean, evaluate whether a response time meets an SLA, or produce any numeric output. Every number in a certification report is the output of Python code, not a language model. This makes numeric results fully reproducible and auditable.

---

## Statistics Computed

For continuous metrics (TTD, TTM, reasoning scores, hallucination scores, token counts), the pipeline computes:

| Statistic | What It Tells You |
|---|---|
| **Mean** | Average across all runs for a fault category |
| **Median** | Middle value — less sensitive to outlier runs |
| **P95** | 95th percentile — what a bad run looks like |
| **Std Dev** | Spread of values — how consistent the agent is |
| **Min / Max** | Full range |
| **Sum** | Used for token counts and raw exposure counts |

P95 is computed directly from sorted values at index `int(n × 0.95)` — no interpolation.

For SLA-normalized timing scores, the same statistics apply to the normalized 0–1 scores. Raw timing values in seconds are also reported separately (mean_s, median_s, p95_s).

---

## Tail Risk: P95

The 95th percentile answers: what does a bad run look like? It is reported alongside mean and median for all timing metrics. A gap between the mean and P95 signals that the agent has occasional severe outliers worth investigating.

---

## Significance Level: α = 0.05

Phase E (optional hypothesis framework, requires n ≥ 30) runs 9 statistical tests at α = 0.05. With multiple tests, expect roughly one borderline result by chance. The pipeline reports individual p-values and effect sizes — the evidence, not just a verdict.

---

## Minimum Sample Sizes

Results are only as reliable as the number of runs behind them. The pipeline uses run count directly — no formal confidence tier labels, but interpretation should account for sample size:

| Runs | Interpretation |
|---|---|
| < 5 | Directional only — don't draw conclusions |
| 5–19 | Useful for internal assessment |
| ≥ 20 | Certification-grade for standard scenarios (S2) |
| ≥ 30 | Required for Phase E hypothesis testing (S1) |

---

## Derived Rates

Binary outcomes (detected / not detected, compliant / not compliant) are computed as simple fractions: count of runs meeting the condition / total runs. These are reported as decimals (0–1). The report renders them as percentages for readability.

---

## Safety: Worst-Case Propagation

Safety and compliance statuses are not averaged — they propagate. One failure in any run flags the category. For performance metrics, an occasional bad run is a tail-risk concern. For safety, it is a categorical failure.

The hard gates that trigger automatic failure are defined in `3.0-Metrics-Reference.md` under Safety & RAI.
