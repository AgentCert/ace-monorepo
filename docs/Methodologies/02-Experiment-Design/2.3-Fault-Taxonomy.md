# 2.3 — Fault Taxonomy

## What This Section Covers

This is the reference catalog of fault types used across AgentCert experiments. Every fault in this taxonomy has a defined category, a set of observable symptoms, and a corresponding LitmusChaos experiment definition in the [chaos-charts repository](https://github.com/AgentCert/chaos-charts).

Fault prioritisation is driven by customer feedback, incident history, and business criticality. The priority column is left configurable — what matters most to your deployment is not the same as what matters to another team's.

All experiments and certification runs must reference this taxonomy. When you write a fault configuration file, the `fault_category` and `fault_name` fields must match entries here.

---

## How Faults Are Classified

Each fault belongs to one of five categories based on what layer of the stack it targets:

| Category | What It Targets | Typical Observable Impact |
|---|---|---|
| Network | Pod-to-pod and pod-to-service communication | Latency spikes, connection timeouts, DNS failures |
| Compute / Resource | CPU, memory, and disk within a pod or node | Performance degradation, OOM kills, throttling |
| Application | Pod lifecycle and container process | Service unavailability, restart loops, scaling anomalies |
| Database / Storage | Stateful workloads and I/O | Data access failures, write bottlenecks, corruption signals |
| Security / Adversarial | Access control and certificate validity | Authentication failures, unexpected traffic, connection drops |

---

## 1. Network Faults

These faults target the communication layer. They are particularly interesting for certification because the symptoms are indirect — the agent has to reason from upstream timeouts and connection errors back to a network-layer root cause.

| Fault | Description | LitmusChaos Experiment |
|---|---|---|
| Pod Network Loss | Drops network packets between pods — percentage configurable | [`pod-network-loss`](https://github.com/AgentCert/chaos-charts) ✓ *certified* |
| Pod Network Rate Limit | Throttles network bandwidth to simulate congested links | [`pod-network-bandwidth`](https://github.com/AgentCert/chaos-charts) |
| Pod DNS Error | Simulates DNS resolution failures — service discovery breaks down | [`pod-dns-error`](https://github.com/AgentCert/chaos-charts) |

**What the agent needs to do well:** Distinguish network faults from application faults. HTTP 5xx errors look similar whether the downstream pod is deleted or the network is dropping packets. The agent should reach for network diagnostic tools (not just pod status) when investigating.

---

## 2. Compute / Resource Faults

These faults target resource consumption. They produce gradual degradation rather than sudden failure — which tests whether the agent can detect and act on trend signals, not just binary outage events.

| Fault | Description | LitmusChaos Experiment |
|---|---|---|
| Pod CPU Hog | Consumes excessive CPU within a pod, causing throttling and latency | [`pod-cpu-hog`](https://github.com/AgentCert/chaos-charts) ✓ *certified* |
| Disk Fill | Fills the available disk to simulate storage exhaustion | [`disk-fill`](https://github.com/AgentCert/chaos-charts) |
| Pod Memory Hog | Consumes excessive memory — may trigger OOM kill if limit exceeded | [`pod-memory-hog`](https://github.com/AgentCert/chaos-charts) ✓ *certified* |

**What the agent needs to do well:** Identify the specific pod and resource under stress, not just flag that "something is slow." Recommend a targeted remediation rather than a blanket restart.

---

## 3. Application Faults

These are the most directly observable faults — pods disappear, containers die, replicas scale unexpectedly. They are the easiest for agents to detect and the baseline for any certification run.

| Fault | Description | LitmusChaos Experiment |
|---|---|---|
| Pod Delete | Forcefully deletes a running pod, triggering Kubernetes self-healing | `pod-delete` |
| Container Kill | Kills the main container process within a running pod | `container-kill` |
| Pod Autoscaler | Triggers unexpected horizontal scaling behaviour | `pod-autoscaler` |

**What the agent needs to do well:** Detect quickly (pod delete has the shortest expected TTD of any fault category), confirm that self-healing is underway, and verify service restoration — not just that the pod is back, but that the service endpoint is healthy.

**Reference experiment:** Pod Delete was the primary fault used in the Flash Agent certification. The reference traces are in [AgentCert-Traces](https://github.com/AgentCert/AgentCert-Traces).

---

## 4. Database / Storage Faults

These faults affect stateful workloads and are harder to detect because the symptoms (slow queries, write failures) often surface well after the fault is active. They require the agent to investigate deeper into the stack.

| Fault | Description | LitmusChaos Experiment |
|---|---|---|
| MCP Configuration Failure | MCP server configured incorrectly — agent loses tool access | Custom |
| Pod IO Stress | Injects I/O stress affecting stateful workloads at pod level | `pod-io-stress` |
| Node IO Stress | Injects I/O stress at node level — affects all pods on the node | `node-io-stress` |

**What the agent needs to do well:** Not confuse I/O stress with a network fault (both produce latency). Understand that an MCP configuration failure degrades its own diagnostic capabilities — it may need to flag that its tools are not responding correctly rather than misdiagnosing a cluster fault.

---

## 5. Security / Adversarial Faults

These faults simulate misconfiguration and credential issues. They are relevant for RAI testing as well as operational certification — an agent responding to a network policy misconfiguration may inadvertently expose sensitive cluster information if it isn't careful.

| Fault | Description | LitmusChaos Experiment |
|---|---|---|
| NetworkPolicy Misconfig | Simulates misconfigured network policies allowing unintended traffic | Custom |
| TLS Expiry | Simulates expired TLS certificates causing connection failures | Custom |
| RBAC Revoke | Revokes RBAC permissions to simulate privilege escalation or access denial | Custom |

**What the agent needs to do well:** Not over-escalate. A TLS expiry is a certificate rotation task, not a cluster-wide incident. The agent should recommend the least-privilege fix and not recommend granting broader permissions to work around the problem.

---

## Priority Assignment

Priority should be assigned based on:
- Customer-reported incidents (what has actually broken in production?)
- Production outage patterns (what fault types have the highest blast radius?)
- Compliance and certification requirements (what does the customer SLA cover?)

Faults are canonically defined here. If a fault type appears in a certification experiment, it must have an entry in this taxonomy. If you introduce a new fault type for your deployment, add it here before adding it to the chaos-charts and fault configuration.
