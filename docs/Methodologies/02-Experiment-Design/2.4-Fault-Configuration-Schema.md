# 2.4 — Fault Configuration Schema

## What This Section Covers

Every fault experiment has a configuration file. This is the answer key — it tells the certifier what was injected, what the agent should ideally have done, and what the SLA thresholds are. The agent never sees it. The certifier uses it to score everything.

This page documents the schema in full. If you are running a new experiment, use this as your reference when writing the fault configuration JSON.

---

## Overview

The **fault configuration schema** defines the complete specification of a fault injected into the simulation environment during an AgentCert experiment. This schema is stored as a JSON document (`fault_configuration.json`) and is the **single source of truth** for each fault injection run.

When a fault is injected, the fault configuration flows into the trace pipeline via the OTEL Collector, providing the ground truth context needed for metrics extraction. **This schema must not be visible to the agent under test** — it is exclusively consumed by the evaluation infrastructure to assess agent performance against known expectations.

---

## Schema Structure

The fault configuration JSON contains the following top-level sections:

| Field | Type | Description |
| --- | --- | --- |
| `fault_id` | `string` | Unique identifier for this fault configuration (e.g., `litmus-pod-delete-001`) |
| `fault_name` | `string` | Name of the fault being injected (e.g., `pod-delete`). Used to populate `fault_type` during metrics extraction |
| `fault_provider` | `string` | Chaos engineering provider (e.g., `LitmusChaos`) |
| `fault_category` | `string` | Category of the fault (e.g., `pod`, `network`, `stress`) |
| `description` | `string` | Human-readable description of what the fault does |
| `injection_timestamp` | `string` | ISO 8601 timestamp of when the fault is injected. Used as the baseline for computing `time_to_detect` and `time_to_mitigate` |
| `fault_configuration` | `object` | Technical parameters governing the fault injection (see below) |
| `expected_impact` | `object` | Expected impact on the target environment (see below) |
| `ground_truth` | `object` | The ideal agent response — symptoms, goals, remediation steps, ideal actions, and ideal tool usage (see below) |

---

## Fault Configuration Parameters

The `fault_configuration` section specifies the technical details of the injection:

| Field | Type | Description |
| --- | --- | --- |
| `target_service` | `string` | The service targeted by the fault |
| `target_namespace` | `string` | Kubernetes namespace of the target |
| `target_label_selector` | `string` | Label selector to identify target pods (e.g., `app=<target-app>`) |
| `chaos_duration_seconds` | `int` | Duration of the chaos experiment in seconds |
| `chaos_interval_seconds` | `int` | Interval between successive fault actions |
| `force_delete` | `bool` | Whether to force-delete the resource |
| `total_chaos_duration` | `string` | Total chaos duration as a string (e.g., `30s`) |
| `ramp_time_seconds` | `int` | Ramp time before fault takes effect |
| `sequence` | `string` | Execution sequence for multiple targets (`parallel` or `serial`) |
| `pods_affected_percentage` | `int` | Percentage of matching pods affected |

---

## Expected Impact

The `expected_impact` section documents the anticipated effect of the fault:

| Field | Type | Description |
| --- | --- | --- |
| `service_disruption` | `bool` | Whether service disruption is expected |
| `affected_resource_type` | `string` | Type of Kubernetes resource affected (e.g., `Pod`) |
| `expected_downtime_seconds` | `int` | Expected downtime duration |
| `expected_recovery_mechanism` | `string` | How the system is expected to self-heal |

---

## Ground Truth

The `ground_truth` section is the most critical part of the schema. It defines the **ideal agent behaviour** against which the agent under test is evaluated. The ground truth is never exposed to the agent — it is used solely by the metrics extraction pipeline to compute accuracy, correctness, and adherence metrics.

### Fault Description, Goal & Remediation

The `fault_description_goal_remediation` sub-section contains:

| Field | Type | Description |
| --- | --- | --- |
| `symptoms` | `List[string]` | Observable symptoms the agent should detect (e.g., pod entering Terminating state, HTTP 5xx errors) |
| `goal` | `string` | The objective of the fault injection — what disruption it introduces and what recovery it triggers |
| `remediation` | `string` | The expected remediation steps the agent should recommend or perform |

### Ideal Course of Action

The `ideal_course_of_action` is an ordered list of steps representing the optimal diagnostic and remediation workflow:

| Field | Type | Description |
| --- | --- | --- |
| `step` | `int` | Step number in the ideal sequence |
| `action` | `string` | Short description of the action |
| `detail` | `string` | Detailed explanation of what the step involves |

This sequence is used to evaluate the agent's **plan adherence**, **action correctness**, and **trajectory efficiency** — comparing the agent's actual actions against this ideal path.

### Ideal Tool Usage Trajectory

The `ideal_tool_usage_trajectory` defines the optimal sequence of tool invocations:

| Field | Type | Description |
| --- | --- | --- |
| `step` | `int` | Step number in the ideal tool sequence |
| `tool` | `string` | Name of the tool/command |
| `command` | `string` | The exact command with placeholder arguments |
| `purpose` | `string` | Why this tool call is expected at this step |

This trajectory is used to compute **tool selection accuracy**, **argument accuracy**, **action efficiency**, and **optimal tool-call deviations** during metrics extraction.

---

## How the Schema Flows into Metrics Extraction

During a fault injection experiment, the following fields from the fault configuration are used by the metrics extraction pipeline (documented in [metrics_extraction_from_traces](metrics_extraction_from_traces)):

1. **`fault_name`** → Populates the `fault_type` metric in the quantitative extraction. The LLM is provided the expected fault type and compares it against what the agent detected.

2. **`injection_timestamp`** → Used as the baseline for computing:
   - `time_to_detect` = `agent_fault_detection_time` − `injection_timestamp`
   - `time_to_mitigate` = `agent_fault_mitigation_time` − `injection_timestamp`

3. **`ground_truth`** → Provides the reference data for evaluating:
   - **`tool_selection_accuracy`**: Were the right tools selected vs. the ideal tool trajectory?
   - **`action_correctness`**: Did the agent's actions match the ideal course of action?
   - **`argument_accuracy`**: Were the correct arguments passed to tools?
   - **`action_efficiency`**: Ratio of effective actions to total actions taken
   - **`optimal_toolcall_deviations`**: How many deviations from the ideal tool usage path
   - **`plan_adherence`**: Did the agent follow a systematic workflow aligned with the ideal course of action?
   - **`acceptance_criteria_met`**: Did the agent correctly detect the anomaly and identify the root cause as defined in the ground truth?

---

## Visibility Constraint

> **The fault configuration schema must never be exposed to the agent under test.** The agent must detect faults, diagnose root causes, and remediate issues based solely on its own observations and tool interactions. The ground truth exists only for post-hoc evaluation by the metrics extraction pipeline.

This separation ensures that certification results reflect the agent's genuine diagnostic and remediation capabilities, not its ability to read pre-supplied answers.

## Reference Fault Configurations

The three faults used in the Flash Agent certification experiment are documented below. These are the live ground truth configurations from [AgentCert-Traces](https://github.com/AgentCert/AgentCert-Traces). The LitmusChaos experiment definitions for all three are in [chaos-charts](https://github.com/AgentCert/chaos-charts).

---

### pod-cpu-hog

**Goal:** Inject CPU stress on target pod containers to validate application resilience and auto-scaling behaviour under high CPU load.

**Symptoms:**
- High CPU usage on targeted pod containers
- Application latency spikes or request timeouts
- Pod CPU throttling observed in metrics
- Warning events related to resource pressure in namespace
- Container CPU usage reaches 100% on stressed cores

**Remediation:** Identify pods with abnormal CPU consumption, kill stress processes inside affected containers, restart degraded pods, and adjust CPU limits or HPA thresholds to handle future CPU pressure.

**Ideal Course of Action:**

| Step | Action | Detail |
|---|---|---|
| 1 | List cluster events | List Kubernetes events across all namespaces to detect warnings related to CPU pressure, throttling, or pod scheduling failures |
| 2 | List pods in affected namespace | Identify pods in degraded, CrashLoopBackOff, or unhealthy state due to CPU exhaustion |
| 3 | Get affected pod details | Retrieve full spec and status to confirm CPU resource limits, requests, and current conditions |
| 4 | Check pod resource consumption | Use metrics to identify which pods have abnormally high CPU consumption |
| 5 | Check node resource consumption | Get node-level resource usage to determine if CPU hog is causing node-wide contention |
| 6 | Exec into pod to inspect processes | Execute `top` or `ps aux` inside the affected container to identify the stress process |
| 7 | Get pod logs | Retrieve logs to identify application-level errors caused by CPU starvation |
| 8 | Query Prometheus for CPU metrics | PromQL: `rate(container_cpu_usage_seconds_total{namespace="<ns>", pod="<pod>"}[5m])` |
| 9 | Kill stress process in container | Exec and kill the stress-ng or md5sum process to stop CPU hog injection |
| 10 | Delete degraded pods | Delete pods stuck in unhealthy state so the controller can recreate them cleanly |
| 11 | Verify deployment health | Confirm desired replicas are running and rollout is healthy after remediation |
| 12 | Adjust CPU limits or scaling policy | Update workload spec to adjust CPU limits or configure HPA for future pressure |

**Ideal Tool Trajectory:**

| Step | Tool | Command |
|---|---|---|
| 1 | Events: List | `events list --all-namespaces` |
| 2 | Pods: List in Namespace | `pods list --namespace <target-namespace>` |
| 3 | Pods: Get | `pods get --namespace <ns> --name <pod>` |
| 4 | Pods: Top | `pods top --namespace <target-namespace>` |
| 5 | Nodes: Top | `nodes top` |
| 6 | Pods: Exec | `pods exec --namespace <ns> --name <pod> --command 'top -bn1'` |
| 7 | Pods: Log | `pods log --namespace <ns> --name <pod>` |
| 8 | Execute PromQL Query | `rate(container_cpu_usage_seconds_total{namespace="<ns>", pod="<pod>"}[5m])` |
| 9 | Pods: Exec | `pods exec --namespace <ns> --name <pod> --command 'kill -9 $(pgrep -f stress)'` |
| 10 | Pods: Delete | `pods delete --namespace <ns> --name <pod>` |
| 11 | Resources: Get | `resources get --apiVersion apps/v1 --kind Deployment --namespace <ns> --name <deploy>` |
| 12 | Resources: Create or Update | Update deployment CPU limits |
| 13 | Configure HPA *(optional)* | `kubectl autoscale deployment/<deploy> --cpu-percent=70 --min=2 --max=10` |
| 14 | Restart Deployment *(optional)* | `kubectl rollout restart deployment/<deploy> --namespace <ns>` |

### pod-memory-hog

**Goal:** Inject memory stress on target pod containers to validate application resilience under high memory load and OOM-kill scenarios.

**Symptoms:**
- High memory usage on targeted pod containers
- OOMKilled or CrashLoopBackOff status on affected pods
- Application latency spikes or request failures
- Warning events related to memory pressure in namespace
- Container memory usage nearing or exceeding limits

**Remediation:** Identify pods with abnormal memory consumption, kill stress processes inside affected containers, restart OOMKilled pods, and adjust memory limits or VPA policies to handle future memory pressure.

**Ideal Course of Action:**

| Step | Action | Detail |
|---|---|---|
| 1 | List cluster events | Detect OOMKilled warnings, memory pressure, and pod eviction events |
| 2 | List pods in affected namespace | Identify pods in OOMKilled, CrashLoopBackOff, or Evicted state |
| 3 | Get affected pod details | Confirm memory resource limits, requests, and last termination reason |
| 4 | Check pod resource consumption | Identify which pods have abnormally high memory consumption |
| 5 | Check node resource consumption | Determine if memory hog is causing node-wide pressure or evictions |
| 6 | Exec into pod to inspect processes | Execute `top` or `ps aux` to identify the stress process consuming memory |
| 7 | Get pod logs | Identify application-level errors caused by memory starvation (OOM, crashes) |
| 8 | Query Prometheus for memory metrics | PromQL: `container_memory_working_set_bytes{namespace="<ns>", pod="<pod>"}` |
| 9 | Kill stress process in container | Exec and kill the stress-ng process to stop memory hog injection |
| 10 | Delete degraded pods | Delete OOMKilled or CrashLoopBackOff pods so controller recreates them cleanly |
| 11 | Verify deployment health | Confirm desired replicas are running and rollout is healthy |
| 12 | Adjust memory limits or scaling policy | Update workload spec or configure VPA for future memory pressure |

**Ideal Tool Trajectory:**

| Step | Tool | Command |
|---|---|---|

| 7 | Pods: Log | `pods log --namespace <ns> --name <pod>` |

### pod-network-loss

**Goal:** Inject network packet loss on target pods to validate application resilience to degraded network conditions, retry logic, circuit breakers, and failover mechanisms.

**Symptoms:**
- High packet loss or dropped connections to target pods
- HTTP 5xx errors and request timeouts from dependent services
- Increased latency and retransmissions on pod network interface
- Warning events on target pods indicating network degradation
- Service mesh or load balancer health checks failing for affected pods
- Application logs showing connection timeouts and socket errors

**Remediation:** Identify pods experiencing packet loss. Inspect pod network configuration and container states. Delete affected pods to restore clean network state. Verify deployment health and confirm network connectivity resumes after chaos ends.

**Ideal Course of Action:**

| Step | Action | Detail |
|---|---|---|
| 1 | List cluster events | Detect network-related warnings, pod failures, or unhealthy endpoints |
| 2 | List pods in affected namespace | Identify pods in degraded, Error, or CrashLoopBackOff state due to network loss |
| 3 | Get affected pod details | Inspect container states, restart counts, and network configuration |
| 4 | Get pod logs | Identify connection timeouts, socket resets, and I/O errors |
| 5 | Exec into pod — network diagnostics | Run `ping` and `curl` inside pod to confirm packet loss on network interface |
| 6 | Check pod resource consumption | Correlate network issues with any resource pressure |
| 7 | Check node-level network stats | Get node stats to identify abnormal transmit/receive error counts |
| 8 | Query Prometheus — network errors | PromQL: `sum(rate(container_network_transmit_errors_total{namespace="<ns>"}[5m])) by (pod)` |
| 9 | Query Prometheus — HTTP error rate | PromQL: `sum(rate(http_requests_total{namespace="<ns>", code=~"5.."}[5m]))` |
| 10 | Delete affected pods | Delete error-state pods so controller recreates them with clean network config |
| 11 | Verify deployment health | Confirm desired replicas are running after remediation |
| 12 | Scale deployment if needed | Ensure sufficient healthy replicas are available after network recovery |
| 13 | Validate network connectivity restored | Exec `ping`/`curl` to confirm packet loss has ceased |
| 14 | Confirm application recovery via Prometheus | Verify HTTP error rate and latency have returned to baseline |

**Ideal Tool Trajectory:**

| Step | Tool | Command |
|---|---|---|
| 1 | Events: List | `events list --all-namespaces` |
| 2 | Pods: List in Namespace | `pods list --namespace <target-namespace>` |
| 3 | Pods: Get | `pods get --namespace <ns> --name <pod>` |
| 4 | Pods: Log | `pods log --namespace <ns> --name <pod>` |
| 5 | Pods: Exec | `pods exec --namespace <ns> --name <pod> --command 'ping -c 4 <svc> && curl -s -o /dev/null -w %{http_code} <svc-url>'` |
| 6 | Pods: Top | `pods top --namespace <target-namespace>` |
| 7 | Node: Stats Summary | `node stats summary --name <node-name>` |
| 8 | Execute PromQL Query | `sum(rate(container_network_transmit_errors_total{namespace="<ns>"}[5m])) by (pod)` |
| 9 | Execute PromQL Query | `sum(rate(http_requests_total{namespace="<ns>", code=~"5.."}[5m]))` |
| 10 | Pods: Delete | `pods delete --namespace <ns> --name <pod>` |
| 11 | Resources: Get | `resources get --apiVersion apps/v1 --kind Deployment --namespace <ns> --name <deploy>` |
| 12 | Resources: Scale | `resources scale --apiVersion apps/v1 --kind Deployment --namespace <ns> --name <deploy> --scale <n>` |
| 13 | Pods: Exec | `pods exec --namespace <ns> --name <pod> --command 'ping -c 4 <target-service>'` |
| 14 | Execute PromQL Query | `sum(rate(http_requests_total{namespace="<ns>", code=~"5.."}[5m]))` |
