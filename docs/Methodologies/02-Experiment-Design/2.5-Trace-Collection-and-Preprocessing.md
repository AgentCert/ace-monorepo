# 2.5 — Trace Collection and Preprocessing

## What This Section Covers

Before the certifier can measure anything, it needs complete, well-structured traces. This page explains how traces are produced during an experiment — what the simulation environment looks like, how events from three different sources converge into a single Langfuse trace, and how that trace is preprocessed into per-fault buckets before metrics extraction begins.

---

## Simulation Environment Components

The simulation environment is composed of five key components:

| Component | Description |
|-----------|-------------|
| **Application Pod** | Runs the target application. Produces its own application-level logs (HTTP requests, errors, latency, etc.). |
| **Fault Pod** | Injects fault configurations into the system. Writes the fault configuration as events into the Langfuse trace via the OTEL Collector. The fault configuration is **not** visible to the Agent Pod. |
| **Agent Pod** | Contains the autonomous agent responsible for identifying faults and performing mitigation actions. Operates independently with no access to fault configuration. Produces its own agent-level logs. The Agent calls the LLM Gateway **directly** for reasoning and invokes tools via the MCP Server for actions. |
| **Tool Calls (MCP Server)** | Provides the tools the Agent invokes to inspect and remediate the system. Agent-Tool interactions are captured via the OTEL Collector. |
| **LLM Gateway** | Hosts multiple LLM models. The Agent calls the LLM Gateway directly. The gateway reports LLM interaction traces **directly** to Langfuse. |

---

## 2. Trace Collection Pipeline

Traces converge into Langfuse from two distinct paths:

| Collection Path | Source | What It Captures |
|----------------|--------|-----------------|
| **Direct (LLM Gateway -> Langfuse)** | LLM Gateway | All LLM interaction traces — prompts, completions, token usage, latency, model selection. |
| **OTEL Collector -> Langfuse** | Fault Pod, Agent Pod (tool calls) | Fault configuration events (injected at experiment start) and Agent-to-Tool interaction events. |

### Why Two Paths?

- The LLM Gateway already has native Langfuse integration, so LLM traces flow directly without an intermediary.
- Fault injection metadata (from Fault Pod) and Agent-Tool interactions (from MCP Server) do not have native Langfuse integration. They are instrumented via OpenTelemetry and routed through an OTEL Collector that writes events into the same Langfuse trace.

**Reading the diagram:**
- The **Agent Pod** has two outgoing connections: one to the **MCP Server** (for tool calls) and one directly to the **LLM Gateway** (for reasoning).
- The **OTEL Collector** receives events from two sources: the **Fault Pod** (fault configuration events) and the **MCP Server** (agent-tool interaction events). Both flow into Langfuse as PATH 2.
- The **LLM Gateway** sends LLM traces directly to Langfuse (PATH 1) — no OTEL intermediary needed.
- At experiment completion, **Langfuse** holds the unified trace containing all three event types.

## 4. Experiment Execution Flow

An experiment proceeds through the following sequence:

1. **Fault Injection Starts** — The Fault Pod injects one or more fault
   configurations into the target Application Pod. The fault configuration
   is simultaneously published as an event to the OTEL Collector, which
   forwards it to Langfuse.

2. **Agent Observes Symptoms** — The Agent Pod (with **no visibility** into
   the fault configuration) monitors the Application Pod through its own
   logs and available tooling.

3. **Agent Reasons and Acts** — The Agent calls the LLM Gateway **directly**
   for reasoning (these interactions are traced directly to Langfuse by the
   gateway). The Agent also invokes tools via the MCP Server (these
   Agent-Tool interactions are captured by the OTEL Collector and forwarded
   to Langfuse).

4. **Traces Converge in Langfuse** — By the end of the experiment, Langfuse
   contains a unified trace comprising:
   - The injected **fault configuration** (ground truth)
   - All **Agent-to-Tool interactions** (actions taken)
   - All **Agent-to-LLM interactions** (reasoning steps)

## 5. Preprocessing Assumptions

Before metrics extraction or aggregation, the following assumptions hold:

| # | Assumption | Rationale |
|---|-----------|-----------|
| 1 | **Langfuse contains the complete trace** | All three event categories (fault config, tool calls, LLM interactions) are present for every experiment run. |
| 2 | **Fault configuration is ground truth** | The fault config injected by the Fault Pod is the authoritative record of what faults were introduced. |
| 3 | **Agent operates without fault visibility** | The Agent Pod has no access to fault configuration — it must independently detect and mitigate issues. |
| 4 | **Multiple faults per experiment** | Experiments inject multiple faults simultaneously to simulate realistic, compounded application failures. |
| 5 | **Temporal ordering is preserved** | Events in the Langfuse trace retain accurate timestamps, allowing reconstruction of the sequence: fault injection -> detection -> mitigation. |
| 6 | **Trace correlation is handled by OTEL + Langfuse** | The OTEL Collector and Langfuse natively correlate events from different sources into a single experiment trace. |
