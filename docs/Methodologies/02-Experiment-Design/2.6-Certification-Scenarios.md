# 2.6 — Certification Scenarios

## What This Page Covers

The certification output depends on how many valid runs you have and whether the pipeline stages complete successfully. This page documents the four scenarios you will encounter and what each one means for the report you receive.

The threshold for full statistical certification is **n ≥ 30 successful runs per fault category**. Everything below that threshold produces a partial or null result.

## S1 — Full Statistical Certification

**Trigger:** n ≥ 30 successful runs per fault category.

This is the complete certification. All three pipeline phases run — Aggregation, Hypothesis Testing (H-01 through H-09), and Certification Report. The output is a certificate with statistical claims that can be defended quantitatively.

**What you get:**
- Detection rate with Wilson confidence interval and a certified lower-bound floor
- TTD and TTM with IQM, P95, and bootstrap BCa confidence intervals
- SLA compliance result from the Wilcoxon one-sample test (H-05)
- Cross-category comparison (H-03) if you have multiple fault categories with n ≥ 30
- Temporal drift assessment (H-09) — is the agent's performance stable across the run sequence?
- Full 12-section certification report including Appendix A1–A5

**When this is the right target:** Always. S1 is the only scenario that produces a statistically defensible certification. If you are certifying an agent for production deployment, you need S1.

---

## S2 — Aggregation-Only Report

**Trigger:** 1–29 successful runs. Not enough for hypothesis testing, but enough to aggregate.

The certifier runs Aggregation and Certification but skips Phase E entirely. The report describes what happened across the runs you have, but cannot make statistical claims about reliability or SLA compliance.

**What you get:**
- Aggregated detection rate and response times (descriptive statistics only — no confidence intervals of meaningful width)
- Qualitative narrative from the LLM Council across however many runs you have
- Safety assessment with worst-case propagation (this still applies regardless of run count)
- A certification report marked as "partial — insufficient runs for statistical validation"

**What you do not get:**
- Confidence intervals (too wide to be useful below 30)
- Hypothesis test results (H-01 through H-09 all require n ≥ 30)
- Statistical lower-bound floor on detection rate

**When this happens:** Early in an experiment when you are still collecting runs, or when some runs were discarded as invalid. S2 is a progress report, not a final certification.

**What to do:** Collect more valid runs until you reach 30. Do not treat an S2 report as a certification — it is a partial view.

---

## S3 — No Certificate

**Trigger:** Zero successful runs reach the certifier. All runs failed upstream — trace is missing, agent crashed, pipeline error before metrics extraction.

Nothing aggregates, nothing certifies. The pipeline exits cleanly and reports the upstream failure.

**What you get:**
- A pipeline failure report indicating that no valid runs were found
- Diagnostic information about why runs failed (if available from trace ingestion)

**What to do:** This is a setup problem, not an agent performance problem. Check:
- Are traces being written to Langfuse correctly?
- Is the fault config being emitted as a trace event?
- Are the Phase 0 and Phase 1 pipeline steps completing without error?

Fix the upstream issue and re-run. Do not interpret S3 as a statement about agent capability — you haven't measured anything yet.

---

## S4 — Certificate with Warnings

**Trigger:** One or more runs completed successfully and trace exists, but metrics extraction returned null fields — the LLM could not extract values from the trace content.

This is the trickiest scenario. Runs happened, the agent operated, but the certifier couldn't score them. The pipeline produces a report but flags every null field explicitly with a warning.

**What you get:**
- A certification report with populated structural sections (methodology, fault taxonomy, pipeline summary)
- Null or placeholder values for quantitative metrics (TTD, TTM, detection rate)
- A warning banner on every affected section: "metrics not extracted — investigate trace quality"
- Qualitative sections may still be populated if the LLM Council could work from partial traces

**Why this happens:**
- The trace format changed (new agent version, new Langfuse schema)
- The LLM extractor encountered traces that didn't match the expected structure
- The fault config didn't flow into the trace correctly (so TTD/TTM baselines are missing)

**What to do:** Inspect the raw traces in the AgentCert-Traces repository. Check whether fault config events are present and structured correctly. If the trace format changed, the Phase 1 extraction prompt or parser may need updating.

Do not treat S4 as an agent failure — it means the measurement infrastructure has an issue, not the agent.

---

## Choosing the Right Response to Each Scenario

| Scenario | Certificate Produced? | Statistical Claims Valid? | Recommended Action |
|---|---|---|---|
| S1 | Yes | Yes | Review report; certify for production if results meet threshold |
| S2 | Partial | No | Collect more runs; treat as interim view only |
| S3 | No | No | Debug upstream pipeline; fix trace collection before re-running |
| S4 | Yes (with warnings) | No | Inspect trace quality; fix extraction before treating report as valid |
