# 4.0 — Pipeline Architecture

The AgentCert pipeline takes a raw Langfuse trace and produces a certification-ready report. This page shows the full flow from Phase 0 through Phase 3, with enough detail to understand what each phase does and why it's designed that way.

---

## The Full End-to-End Flow

```
┌─────────────────────────────────────────────────────────────────────────┐
│  INPUT: Raw Langfuse Trace JSON                                         │
│                                                                         │
│  A single JSON file containing all spans from one agent run.            │
│  Events from multiple concurrent faults are interleaved.               │
│  One trace file = one run against one set of injected faults.          │
└─────────────────────────────────────────────────────────────────────────┘
                                   │
                                   ▼
╔═════════════════════════════════════════════════════════════════════════╗
║  PHASE 0 — FAULT BUCKETING                          fault_analyzer/    ║
║                                                                         ║
║  Challenge: the trace is a stream of interleaved events from           ║
║  multiple simultaneous faults. We need to separate them.               ║
║                                                                         ║
║  Algorithm (3 passes):                                                  ║
║   1. Scan spans for "fault: *" tags → create one bucket per fault      ║
║      with injection timestamps (deterministic, no LLM)                 ║
║   2. Extract run-level metadata: agent_id, experiment_id, run_id       ║
║   3. For remaining events, classify into fault buckets:                ║
║       • 0 faults active at this timestamp → unclassified               ║
║       • 1 fault active → assign deterministically (no LLM call)        ║
║       • 2+ faults active → send to LLM classifier in batches           ║
║                                                                         ║
║  LLM classifier uses:                                                   ║
║   • System-role caching (~50% token rebate on batches 2..N)            ║
║   • Compact fault context (84% smaller prompt than verbose)            ║
║   • Structured Pydantic output: related_faults + confidence            ║
║                                                                         ║
║  Output files:                                                          ║
║   • {stem}_bucket_{fault_name}.json    (one per fault)                 ║
║   • {stem}_bucketing_manifest.json     (summary + token counts)        ║
║   • {stem}_unclassified.json           (events not assigned)           ║
╚═════════════════════════════════════════════════════════════════════════╝
                                   │
                                   │  (one bucket file per fault)
                                   ▼
╔═════════════════════════════════════════════════════════════════════════╗
║  PHASE 1 — METRICS EXTRACTION                    metrics_extractor/    ║
║                                                                         ║
║  For each fault bucket from a single run:                              ║
║                                                                         ║
║  Batch processing:                                                      ║
║   • Sort spans chronologically                                          ║
║   • Split into batches of ~6 spans                                     ║
║   • For each batch: LLM extracts raw observations                      ║
║                                                                         ║
║  What the LLM does (text only):                                        ║
║   • Extract timestamps: fault detection time, mitigation time          ║
║   • Identify: fault type, target service, namespace                    ║
║   • Assess: reasoning quality, plan adherence, RAI compliance          ║
║   • Count: PII instances, hallucinations, adversarial inputs           ║
║                                                                         ║
║  What Python code does (all arithmetic):                               ║
║   • TTD = detection_time − injection_timestamp                         ║
║   • TTM = mitigation_time − injection_timestamp                        ║
║   • hallucination_score = hallucination_count / total_response_count   ║
║   • Sum token counts, PII counts, tool call counts across batches      ║
║   • Average reasoning scores, compute boolean OR for PII detection     ║
║                                                                         ║
║  Hallucination types tracked (combined_judge):                         ║
║   • UNGROUNDED: claims not supported by tool outputs                   ║
║   • FABRICATED_TOOL_CALL: invented tool calls or results               ║
║   • TRAJECTORY_DEVIATION: diverged from ideal tool trajectory          ║
║   • IGNORED_ERROR: error signals dismissed without investigation       ║
║                                                                         ║
║  Output → ExtractionResult → MongoDB (one document per run)            ║
╚═════════════════════════════════════════════════════════════════════════╝
                                   │
                                   │  Repeat ~30 runs × N faults × M fault categories
                                   ▼
╔═════════════════════════════════════════════════════════════════════════╗
║  PHASE 2 — AGGREGATION                                 aggregator/     ║
║                                                                         ║
║  Query all per-run documents for a fault category from MongoDB.        ║
║                                                                         ║
║  Numeric aggregation (deterministic Python):                           ║
║   • Continuous metrics (TTD, TTM, reasoning score):                    ║
║     IQM, mean, median, P95, P99, bootstrap CI (BCa, B=10,000)         ║
║   • Rate metrics (detection rate, mitigation rate, SLA compliance):   ║
║     Wilson score CI — gives a certified lower-bound floor              ║
║   • Count metrics (tokens, PII instances):                             ║
║     Sum across all runs                                                 ║
║                                                                         ║
║  Narrative aggregation (LLM Council):                                  ║
║   • k independent judges each read all ~30 narratives for one metric   ║
║   • Each judge produces: summary + severity label + confidence         ║
║   • A meta-judge reconciles all k outputs into one consensus           ║
║   • Current config: council_size=1, extendable to 3-5 judges          ║
║                                                                         ║
║  Safety aggregation (rates, not worst-case):                          ║
║   • RAI hard gate: if personal_pii_detected=True or                   ║
║     adversarial_input_count>0 in any run → that run's RAI score = 0  ║
║   • All other safety fields (security compliance, bias, guardrails)   ║
║     are aggregated as rates — fraction of runs that passed            ║
║                                                                         ║
║  RAI scoring:                                                           ║
║   • Security compliance rate × PII clean rate × adversarial clean     ║
║   • Fairness score: cross-category consistency via LLM Council         ║
║                                                                         ║
║  Output → per-category CertificationScorecard → MongoDB                ║
╚═════════════════════════════════════════════════════════════════════════╝
                                   │
                                   ▼
╔═════════════════════════════════════════════════════════════════════════╗
║  PHASE 3 — CERTIFICATION                               cert_builder/   ║
║                                                                         ║
║  3.1 INGESTION                                                          ║
║   Parse raw scorecards into structured ParsedContext                   ║
║   Extract metadata: agent_name, date, total_runs, categories           ║
║                                                                         ║
║  3.2 COMPUTATION (6 builders, no LLM calls)                            ║
║   • ScoreCard builder: 8-dimension radar (Reliability, Safety,         ║
║     Resilience, Resource Efficiency, Reasoning, Hallucination,         ║
║     Security, RAI Fairness) — each dimension 0–1                       ║
║   • Table builder: detection/mitigation rates with CI whiskers         ║
║   • Chart builder: radar, bar, line chart data (depends on scorecard)  ║
║   • Assessment builder: resilience/security/performance ratings        ║
║   • Card builder: key-value metadata blocks                            ║
║   • Hardcoded loader: static YAML (methodology, glossary)              ║
║                                                                         ║
║  3.3 NARRATIVES (7 LLM builders — 6 concurrent, 1 sequential)          ║
║   Concurrent:                                                           ║
║    ├── Scope & methodology narrative                                    ║
║    ├── Key findings summary                                             ║
║    ├── Qualitative metrics interpretation                               ║
║    ├── Fault analysis (per category, via LLM Council)                  ║
║    ├── Limitations narrative                                           ║
║    └── Fairness score synthesis                                        ║
║   Sequential (depends on limitations):                                 ║
║    └── Recommendations                                                 ║
║                                                                         ║
║  3.4 ASSEMBLY                                                           ║
║   Merge 3.1 + 3.2 + 3.3 → 12-section CertificationReport             ║
║   Pydantic v2 validation — pipeline errors rather than emitting        ║
║   a malformed report                                                    ║
║                                                                         ║
║  Output → certification_report.json → HTML + PDF                       ║
╚═════════════════════════════════════════════════════════════════════════╝
                                   │
                                   ▼ (optional, n ≥ 30 per fault category)
╔═════════════════════════════════════════════════════════════════════════╗
║  PHASE E — HYPOTHESIS FRAMEWORK            hypothesis_framework/       ║
║                                                                         ║
║  9 tests applied to the same per-run metrics:                          ║
║   H-01  Confidence intervals (IQM + Bootstrap BCa)                     ║
║   H-02  Success rate estimation (Wilson CI, certified floor)           ║
║   H-03  Cross-category comparison (Kruskal-Wallis + Mann-Whitney)      ║
║   H-04  Success rate uniformity (Fisher's Exact)                       ║
║   H-05  Variance stability (Levene's + CV)                             ║
║   H-06  SLA threshold compliance (Wilcoxon one-sample)                 ║
║   H-07  SLA breach rate (Exact Binomial)                               ║
║   H-08  Tail risk analysis (CVaR, expected overshoot)                  ║
║   H-09  Temporal stability (CUSUM, EWMA, change point detection)       ║
║                                                                         ║
║  H-01 to H-05 always run.                                              ║
║  H-06 to H-07 require SLA thresholds.                                  ║
║  H-08 to H-09 always run; add threshold-specific analysis with SLAs.  ║
║                                                                         ║
║  Output → injected as Appendix A1–A5 in the certification report       ║
╚═════════════════════════════════════════════════════════════════════════╝
```

## Phase Summary Table

| Phase | Module | Input | Output | Key Design Decision |
|---|---|---|---|---|
| 0 — Fault Bucketing | `fault_analyzer/` | Raw Langfuse trace JSON | Per-fault bucket JSON files | Temporal windowing to minimize LLM calls; system-role caching on classifier |
| 1 — Metrics Extraction | `metrics_extractor/` | Fault bucket JSON (single run) | ExtractionResult in MongoDB | LLM extracts text; Python does all arithmetic — no LLM math |
| 2 — Aggregation | `aggregator/` | ~30 MongoDB docs per fault category | CertificationScorecard in MongoDB | Deterministic stats for numbers; LLM Council for narratives; worst-case for safety |
| 3 — Certification | `cert_builder/` | CertificationScorecard | certification_report.json (12 sections, Pydantic-validated) | 6 concurrent LLM narrative builders; recommendations waits for limitations |
| E — Hypothesis | `hypothesis_framework/` | Per-run metrics + optional SLA thresholds | Statistical evidence block | 9 tests, 16 methods, grounded in 20 peer-reviewed papers |

---

## Key Design Principles Across All Phases

**LLMs never do arithmetic.** Every numeric computation — sum, ratio, average, percentile, confidence interval — is implemented in Python. The LLM handles only text extraction and narrative synthesis. This matters for reproducibility and for avoiding hallucinated numbers that look plausible but are subtly wrong.

**Statistical rigour through repetition.** Each fault is repeated ~30 times to capture the natural variance in non-deterministic LLM agent outputs. A single run tells you what happened once; 30 runs tell you whether the agent is reliably good or just occasionally lucky.

**LLM Council for textual consensus.** Multiple independent LLM judges produce summaries that are reconciled by a meta-judge. This reduces single-model bias and provides a calibrated confidence signal: when judges agree, the finding is robust; when they disagree, the metric should be flagged for human review.

**Safety metrics are rates, not worst-case flags.** Most safety and compliance fields (security compliance, bias detection, guardrail violations) are aggregated as rates — the fraction of runs where the condition held. The one true exception is the RAI hard gate: if `personal_pii_detected` is True or `adversarial_input_count > 0` in any run, that run's RAI score is zeroed regardless of other metrics. You cannot average away a PII exposure, but you can and should quantify how often security compliance held.

**Storage-agnostic query interface.** The aggregation phase uses the same interface whether reading from a directory of JSON files or a MongoDB collection. The orchestrator doesn't know which backend it's using.
