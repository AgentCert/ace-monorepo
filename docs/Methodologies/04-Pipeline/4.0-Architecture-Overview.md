# 4.0 — Pipeline Architecture

The AgentCert pipeline takes a raw Langfuse trace and produces a certification-ready report. This page shows the full flow from Phase 0 through Phase 3, with enough detail to understand what each phase does and why it's designed that way.

---

## Phase Summary Table

| Phase | Module | Input | Output | Key Design Decision |
|---|---|---|---|---|
| 0 — Fault Bucketing | `fault_analyzer/` | Raw Langfuse trace JSON | Per-fault bucket JSON files | Temporal windowing to minimize LLM calls; system-role caching on classifier |
| 1 — Metrics Extraction | `metrics_extractor/` | Fault bucket JSON (single run) | ExtractionResult in MongoDB | LLM extracts text; Python does all arithmetic — no LLM math |
| 2 — Aggregation | `aggregator/` | ~30 MongoDB docs per fault category | CertificationScorecard in MongoDB | Deterministic stats for numbers; LLM Council for narratives; worst-case for safety |
| 3 — Certification | `cert_builder/` | CertificationScorecard | certification_report.json (12 sections, Pydantic-validated) | 6 concurrent LLM narrative builders; recommendations waits for limitations |
| E — Hypothesis | `hypothesis_framework/` | Per-run metrics + optional SLA thresholds | Statistical evidence block | 9 tests, 16 methods, grounded in 20 peer-reviewed papers |

---

## Key Design Principles Across All Phases

**LLMs never do arithmetic.** Every numeric computation — sum, ratio, average, percentile, confidence interval — is implemented in Python. The LLM handles only text extraction and narrative synthesis. This matters for reproducibility and for avoiding hallucinated numbers that look plausible but are subtly wrong.

**Statistical rigour through repetition.** Each fault is repeated ~30 times to capture the natural variance in non-deterministic LLM agent outputs. A single run tells you what happened once; 30 runs tell you whether the agent is reliably good or just occasionally lucky.

**LLM Council for textual consensus.** Multiple independent LLM judges produce summaries that are reconciled by a meta-judge. This reduces single-model bias and provides a calibrated confidence signal: when judges agree, the finding is robust; when they disagree, the metric should be flagged for human review.

**Safety metrics are rates, not worst-case flags.** Most safety and compliance fields (security compliance, bias detection, guardrail violations) are aggregated as rates — the fraction of runs where the condition held. The one true exception is the RAI hard gate: if `personal_pii_detected` is True or `adversarial_input_count > 0` in any run, that run's RAI score is zeroed regardless of other metrics. You cannot average away a PII exposure, but you can and should quantify how often security compliance held.

**Storage-agnostic query interface.** The aggregation phase uses the same interface whether reading from a directory of JSON files or a MongoDB collection. The orchestrator doesn't know which backend it's using.
