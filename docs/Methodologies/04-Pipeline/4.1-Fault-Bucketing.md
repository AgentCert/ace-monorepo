# 4.1 — Fault Bucketing

Before any metrics can be extracted, the raw Langfuse trace needs to be decomposed into per-fault lifecycle buckets. This page explains how that works, why it's designed the way it is, and what you'll find in the output files.

---

## Why Fault Bucketing Exists

A Langfuse trace from an AgentCert experiment contains events from an AI agent operating in a Kubernetes cluster where multiple faults may be active simultaneously. The trace is a flat, time-ordered stream of spans — LLM calls, tool invocations, reasoning steps — with no inherent labelling that says "this event belongs to the pod-delete fault" or "this belongs to the CPU hog fault."

The metrics extraction phase (Phase 1) needs to evaluate the agent's response to each fault independently. It needs to know: when did the agent detect this specific fault? What tools did it use to mitigate it? How long did that take? None of that is answerable without first separating the interleaved events into per-fault buckets.

Fault bucketing is Phase 0 — it runs before extraction, and everything downstream depends on it getting the assignments right.

---

## How the Algorithm Works

The bucketing pipeline runs in three passes over the trace.

```
Raw Langfuse Trace JSON
          │
          ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  PASS 1 — DETERMINISTIC FAULT SPAN DETECTION                            │
│                                                                         │
│  Scan every span in the trace for the pattern "fault: *" in the        │
│  span name or metadata.                                                 │
│                                                                         │
│  For each fault span found:                                             │
│   • Extract: fault_name, injection_timestamp, injection_end_timestamp  │
│   • Extract: target, timing, injection params, probes, workflow        │
│   • Create a FaultBucket (empty container, ready for events)           │
│   • Handle duplicates: if an active bucket already exists for this     │
│     fault_name, skip. If only closed buckets exist, create a new       │
│     one with a numeric suffix (e.g., pod-delete, pod-delete_2)        │
│                                                                         │
│  This pass is entirely deterministic — no LLM involvement.             │
│  Output: N fault buckets with injection windows defined.               │
└─────────────────────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  PASS 2 — METADATA EXTRACTION                                           │
│                                                                         │
│  Scan early trace events to extract run-level metadata:                │
│   • agent_id, agent_version                                            │
│   • experiment_id, run_id                                              │
│   • Any global context that applies to the whole run                   │
│                                                                         │
│  This is a lightweight pass — just collecting provenance information.  │
│  Also extracts run-level token totals ONCE here, before the            │
│  per-fault bucketing loop, to avoid double-counting tokens             │
│  across multiple fault buckets.                                        │
└─────────────────────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  PASS 3 — EVENT CLASSIFICATION                                          │
│                                                                         │
│  For every remaining event (not a fault-definition span):              │
│                                                                         │
│  Step A: Temporal windowing                                             │
│   Check how many fault buckets are active at this event's timestamp.   │
│   "Active" means the fault's injection window contains this timestamp. │
│                                                                         │
│   ┌────────────────┬──────────────────────────────────────────────┐   │
│   │ Active faults  │ Action                                       │   │
│   ├────────────────┼──────────────────────────────────────────────┤   │
│   │ 0              │ Mark as unclassified (no fault context)      │   │
│   │ 1              │ Assign directly — no LLM call needed         │   │
│   │ 2 or more      │ Send to LLM classifier (see below)           │   │
│   └────────────────┴──────────────────────────────────────────────┘   │
│                                                                         │
│  Step B: LLM classification (only when 2+ faults active)               │
│   Events are batched (default batch_size=1) and sent to the LLM        │
│   classifier with:                                                      │
│    • A "known faults" block listing all active faults and their        │
│      injection metadata (compact format: ~84% smaller than verbose)   │
│    • The event batch                                                    │
│    • A list of candidate_fault_ids for this batch                      │
│                                                                         │
│   The LLM returns per-event:                                           │
│    • related_faults: list of fault IDs this event is relevant to      │
│    • confidence: 0–1                                                   │
│    • fault_reasoning: short explanation                                │
│    • fault_detected / fault_mitigated: bool flags                      │
│                                                                         │
│  If the LLM call fails, fallback_classify() assigns the event to       │
│  all active faults with a conservative confidence of 0.3.              │
└─────────────────────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  OUTPUT FILES                                                           │
│                                                                         │
│  {stem}_bucket_{fault_name}.json    ← one per fault                    │
│  {stem}_bucketing_manifest.json     ← summary of all buckets           │
│  {stem}_unclassified.json           ← events not assigned to any fault │
│  {stem}_other_detected_faults.json  ← faults agent found but we       │
│                                        didn't inject (interesting!)    │
│  batch_classification_trace.json    ← (debug mode only) full LLM      │
│                                        routing trace per event         │
│  ground_truth/                      ← per-fault ground_truth.json     │
└─────────────────────────────────────────────────────────────────────────┘
```

## The LLM Classifier in Detail

The classifier (`fault_analyzer/scripts/classifier.py`) uses GPT-4o with structured output. Two features significantly reduce its cost.

### System-Role Caching

Azure OpenAI's prompt cache gives a ~50% token rebate on stable system-role content. The classifier puts the "known faults" block — which is the same for all batches in a run — into the system role. Batch 1 warms the cache. Batches 2 through N hit the cache and cost roughly half as much.

This matters in practice because a trace with many events and multiple simultaneous faults can generate a large number of classifier calls.

### Compact vs. Verbose Fault Context

There are two ways to render the known faults block that appears in every classifier prompt:

| Mode | What It Includes | Size |
|---|---|---|
| **Compact** (default) | Fault name, category, injection timestamp, end timestamp — minimal fields | ~84% smaller than verbose |
| **Verbose** | Full fault metadata including target, timing parameters, probes, workflow | Baseline |

You can toggle between them with `fault_pruning: true/false` in the classifier config. Compact mode is the default because it substantially reduces per-batch token cost with minimal accuracy impact. Use verbose mode when you're debugging classification accuracy on complex overlapping faults.

### Fallback Behavior

If an LLM call fails or returns a malformed response, the classifier falls back to a conservative assignment: the event is copied into all currently active fault buckets with a confidence score of 0.3. This ensures no events are silently dropped, but the low confidence score means downstream analysis can identify and review these assignments.

### LLM Classifier Output Schema

The classifier returns the following fields per event:

| Field | Type | Description |
|---|---|---|
| `related_faults` | list[str] | One or more fault IDs this event relates to — an event can legitimately belong to multiple faults |
| `confidence` | 0–1 float | Classification confidence |
| `fault_reasoning` | dict | Per-fault explanation of why the event was assigned to each fault |
| `fault_detected` | str / null | Fault name the agent first recognized (e.g., `"pod-delete"`). Null if this event is not a detection event. Can be a fault not yet in the known faults list. |
| `detected_fault_severity` | str / null | Severity (critical / high / medium / low) if `fault_detected` is set |
| `detected_fault_target_pod` | str / null | Target pod/resource if `fault_detected` is set |
| `detected_fault_namespace` | str / null | Kubernetes namespace if `fault_detected` is set |
| `detected_fault_signals` | list[str] | Symptoms that led to detection (e.g., `["CrashLoopBackOff"]`) |
| `fault_mitigated` | str / null | Fault name/ID the agent confirmed remediated. Null if not a mitigation event. |
| `has_quantitative_value` | bool | Whether the event contains a measurable numeric value (latency, count, threshold) |
| `has_qualitative_value` | bool | Whether the event contains a subjective or descriptive assessment (severity label, root-cause hypothesis) |
| `has_cost_token_details` | bool | Whether the event contains LLM cost or token usage information |

### Worked Example

Suppose an experiment injects two faults simultaneously:
- **F1** — Pod crash-loop in `payment-service`
- **F2** — Memory limit exceeded in `order-service`

| Event | Related Faults | Detected | Mitigated | Notes |
|---|---|---|---|---|
| `evt-001` | F1 | F1 | — | Agent detects payment-service crash-loop; restart count extracted |
| `evt-002` | F1, F2 | — | — | Agent parses logs relevant to both faults — shared diagnostic step |
| `evt-003` | F2 | F2 | — | Agent detects memory breach in order-service |
| `evt-004` | F1 | — | — | LLM generation reasoning about F1; token details present |
| `evt-005` | F1 | — | F1 | Agent restarts pod — **F1 mitigated**; F1 bucket closed |
| `evt-006` | F2 | — | — | Continued diagnosis for F2 |
| `evt-007` | F2 | — | — | LLM reasoning step for F2 |
| `evt-008` | F2 | — | F2 | Agent scales memory limit — **F2 mitigated**; F2 bucket closed |

After `evt-005`, F1 is removed from the active-faults list and its bucket is frozen. After `evt-008`, same for F2. Note that `evt-002` appears in both buckets — a single event can legitimately belong to multiple faults.

## What's in a Fault Bucket

Each `{stem}_bucket_{fault_name}.json` file is a `FaultBucket` object:

```json
{
  "fault_id": "litmus-pod-delete-001",
  "fault_name": "pod-delete",
  "status": "closed",
  "severity": null,
  "target_pod": "agent-demo",
  "namespace": "default",
  "injection_timestamp": "2026-01-30T17:19:00Z",
  "injection_end_timestamp": "2026-01-30T17:19:30Z",
  "injection_metadata": {
    "timing": { "chaos_duration": "30s" },
    "injection": { "force_delete": false },
    "probes": [...],
    "workflow": "sequential"
  },
  "detected_at": "2026-01-30T17:19:05Z",
  "mitigated_at": "2026-01-30T17:20:26Z",
  "events": [
    {
      "event_id": "span-abc123",
      "type": "GENERATION",
      "name": "agent-reasoning-step",
      "startTime": "2026-01-30T17:19:04Z",
      "related_faults": ["litmus-pod-delete-001"],
      "confidence": 0.92,
      "fault_detected": "litmus-pod-delete-001",
      "fault_mitigated": null,
      "fault_reasoning": {"litmus-pod-delete-001": "Agent identifies pod termination event"}
    }
  ],
  "ground_truth": { ... },
  "sla": { "time_to_detect": 60, "time_to_mitigate": 120 },
  "ideal_course_of_action": [ ... ],
  "ideal_tool_usage_trajectory": [ ... ]
}
```

The `ground_truth`, `sla`, `ideal_course_of_action`, and `ideal_tool_usage_trajectory` fields are copied from the fault configuration schema. They flow into Phase 1 extraction as the reference against which the agent's actual behavior is evaluated.

## The Manifest File

`{stem}_bucketing_manifest.json` is the index across all buckets from this run:

```json
{
  "trace_file": "raw_trace.json",
  "total_faults": 3,
  "total_events_assigned": 398,
  "other_detected_faults_count": 0,
  "unclassified_event_count": 14,
  "trace_tokens": {
    "description": "Tokens extracted ONCE from entire trace (before bucketing) to avoid double-counting",
    "input_tokens": 45230,
    "output_tokens": 1870,
    "total_tokens": 47100
  },
  "llm_tokens_used": {
    "description": "Tokens used by LLM classifier to assign events to fault buckets",
    "input_tokens": 8420,
    "output_tokens": 1140,
    "total_tokens": 9560
  },
  "buckets": [
    {
      "fault_id": "litmus-pod-delete-001",
      "fault_name": "pod-delete",
      "severity": null,
      "target_pod": "agent-demo",
      "namespace": "default",
      "status": "closed",
      "event_count": 134,
      "detected_at": "2026-01-30T17:19:05Z",
      "mitigated_at": "2026-01-30T17:20:26Z",
      "injection_timestamp": "2026-01-30T17:19:00Z",
      "output_file": "trace_bucket_pod-delete.json"
    }
  ]
}
```

The manifest has two token blocks: `trace_tokens` records the agent token usage extracted from the entire trace before bucketing (avoiding double-counting across buckets), while `llm_tokens_used` records what the classifier itself spent on event classification. Both feed into the pipeline-level token accounting in the certification report's Section 11.

## Configuration Reference

`fault_analyzer/config/fault_bucketing_config.json`:

| Parameter | Default | What It Controls |
|---|---|---|
| `classifier.model_name` | `gpt-4o` | Azure OpenAI deployment for classification |
| `classifier.temperature` | `0.1` | Low temperature for consistent classification decisions |
| `classifier.max_tokens` | `4000` | Max response tokens per batch |
| `classifier.fault_pruning` | `false` | `true` = compact fault context (84% smaller); `false` = verbose |
| `classifier.cache_enabled` | `false` | `true` = enable system-role caching for ~50% token rebate |
| `classifier.include_event_input` | `true` | `true` = include both span input and output; `false` = output only |
| `classifier.fallback_confidence` | `0.3` | Confidence score assigned on fallback assignment |
| `pipeline.default_batch_size` | `10` | Default events per LLM classifier call (when not passed via CLI `--batch-size`) |
| `pipeline.max_filename_stem_length` | `80` | Max characters in output filename stems |

## Common Questions

**Why is the default batch size 10?**
The code default (`pipeline.default_batch_size` in config) is 10. You can pass a lower value like `--batch-size 1` when debugging complex traces with many concurrent faults — smaller batches reduce per-call disambiguation burden at the cost of more LLM calls. See Observation 1.2 in `06-Observations` for the measured impact of batch size on classification quality.

**What happens to events that span across multiple fault windows?**
If an event's timestamp falls inside the injection windows of two or more active faults, it goes to the LLM classifier. The classifier can assign it to one, several, or all of the active faults — an event can legitimately belong to multiple buckets.

**What is the "other detected faults" file?**
Sometimes the agent detects a fault that wasn't injected by the experiment — for example, a pre-existing cluster issue. These are collected in `{stem}_other_detected_faults.json`. They're not used in certification scoring but are worth reviewing; they can indicate environmental noise in your test cluster or an overly sensitive agent.

**Can I run bucketing without extraction?**
Yes. Phase 0 and Phase 1 are separate pipeline stages. You can run Phase 0 standalone using `run_bucketing_and_extraction_pipeline.py --batch-size 10` without the `--store` flag to produce bucket files without writing to MongoDB.
