# 5.0 — Certification Overview

The certification phase takes the aggregated scorecards produced by Phase 2 and turns them into a structured 12-section report. This page explains how the cert builder works, what goes into each section of the report, and how the cert reporter renders the final HTML and PDF output.

---

## Overview

There's a temptation to think of certification as "just report generation." In practice, Phase 3 is more interesting than that. It runs six deterministic computation builders, then runs seven LLM narrative builders (six concurrently, one sequentially after limitations), then assembles everything into a Pydantic-validated output. If validation fails, the pipeline errors — it never emits a malformed report.

The output is a `certification_report.json` that the cert reporter converts to HTML and PDF. The report has four logical parts:

| Part | Sections | Focus |
|---|---|---|
| **Agent Capability** | 1–5 | What the agent can do — performance, reasoning, safety |
| **Fault Injection Analysis** | 6–8 | How it behaved across specific fault categories |
| **Limitations & Improvements** | 9–10 | What the agent struggles with and what to do about it |
| **Appendix** | 11–12 | Token accounting and optional statistical hypothesis results |

## The 12 Sections

### Section 1 — Executive Summary

The opening section. Contains an overview of the agent's capabilities, a snapshot of the key metrics, and a findings block with severity-tagged items. This is what a non-technical stakeholder reads to understand whether the agent is ready for production.

The findings block items each carry a severity: `concern` (something that needs fixing), `good` (something that worked well), or `note` (context worth knowing). The LLM key-findings builder generates these; the scorecard builder determines the overall framing.

### Section 2 — Methodology

Explains how the experiment was structured: the fault injection framework (LitmusChaos), the Kubernetes test bed, the ground truth definition process, and the statistical approach. If Phase E (hypothesis framework) was run, this section also explains the statistical testing methodology at a high level.

This section is mostly generated from static YAML loaded by the hardcoded loader, with a short LLM-generated narrative adapting it to the specific experiment.

### Section 3 — Qualitative Findings

Covers the LLM-synthesized qualitative assessment: how the agent reasons, whether it hallucinates, how it adheres to ideal diagnostic plans, and whether its responses are safe and compliant. These assessments come from the qualitative metrics builder in Phase 3.3 and draw on the combined_judge outputs from Phase 1.

### Section 4 — Detection & Response

The quantitative heart of the report. Contains:
- Per-category detection rates with Wilson CI bounds
- Per-category mitigation rates with Wilson CI bounds
- Time-to-detect and time-to-mitigate tables: mean, median, P95 with bootstrap CI whiskers
- SLA compliance status where SLA thresholds are defined
- Bar charts showing rates by fault category

If Phase E ran, this section also includes the H-01 and H-02 statistical findings inline.

### Section 5 — Reasoning Quality

Breaks down the agent's reasoning assessment across four dimensions:
- **Logical coherence**: does the reasoning flow make sense?
- **Diagnostic depth**: does the agent explore the problem systematically?
- **Tool usage relevance**: are the right tools being called for the situation?
- **Explanation clarity**: are the agent's outputs understandable?

These are scored by the combined_judge (Phase 1), averaged across runs (Phase 2), and interpreted by the qualitative narrative builder (Phase 3.3).

### Section 6 — Safety & Security

The safety section. Three components:

1. **RAI score**: weighted composite of security compliance rate, PII clean rate, and adversarial input handling rate
2. **Fairness assessment**: cross-category consistency via LLM Council
3. **Compliance status**: worst-case RAI and security compliance across all runs

This section uses the assessment builder's security card and the fairness_builder's output. A single non-compliant run anywhere causes the category-level status to be flagged.

### Section 7 — Resource Efficiency

Token consumption across the agent's runs: input tokens, output tokens, total tokens, plus retry counts and API call efficiency. This lets you understand what the agent costs per fault and whether certain fault categories trigger significantly more LLM usage than others.

Also includes trajectory step counts (total spans per run) as a proxy for agent verbosity and exploration depth.

### Section 8 — Fault Injection Analysis

The per-category deep-dive section. For each fault category (pod, network, resource, misconfiguration, etc.), this section contains:
- An LLM Council narrative for that category
- Severity assessment and recovery strategy analysis
- Notable limitations specific to this fault type
- Performance delta vs. the overall agent baseline

This is generated by the fault_analysis_builder in Phase 3.3, which runs an LLM Council call per category. The results are assembled into nested sub-sections.

### Section 9 — Limitations

An honest accounting of what the agent struggles with. The limitations builder synthesizes recurring themes from all per-run known_limitations outputs across all fault categories. It identifies patterns: does the agent consistently struggle with network-layer faults? Does it have trouble with multi-pod scenarios? Does it sometimes miss subtle resource exhaustion patterns?

This section is important for customers taking the system forward — it tells you where to focus improvement effort.

### Section 10 — Recommendations

Evidence-based recommendations derived from the limitations analysis. The recommendations builder runs sequentially after the limitations builder (it explicitly reads that output as input). Recommendations are prioritized by impact on the dimensions the agent scored lowest on.

This is not a generic set of suggestions — each recommendation is grounded in specific evidence from the experimental data.

### Section 11 — Pipeline Token Metrics

A full accounting of LLM token usage across all four pipeline phases:

| Phase | What's Counted |
|---|---|
| Phase 0 (Fault Bucketing) | Agent trace tokens + LLM classifier tokens |
| Phase 1 (Metrics Extraction) | LLM tokens for quantitative and qualitative extraction per run |
| Phase 2 (Aggregation) | LLM Council tokens across all categories |
| Phase 3 (Certification) | Narrative builder tokens per builder, summed |

This section exists because certification is not free — multiple LLM calls across 30+ runs per category adds up. Token accounting lets you understand and optimize the pipeline's operational cost.

### Section 12 — Appendix (Optional — Statistical Hypothesis Results)

Appears only when Phase E (hypothesis framework) ran. Contains five sub-sections:

| Appendix | Content |
|---|---|
| A1 — Framework Overview | Description of H-01 through H-09, what each tests and why |
| A2 — Statistical Evidence | Detailed H-01 and H-02 results: CIs, certified floors |
| A3 — Cross-Category Comparison | H-03, H-04, H-05 results: category disparities, uniformity, variance |
| A4 — SLA-Aware Analysis | H-06, H-07, H-08, H-09 results (only when SLA thresholds provided) |
| A5 — Statistical Concepts Glossary | Plain-English explanations of IQM, Wilson CI, CVaR, etc. |

If Phase E did not run (typically because n < 30 per fault category), Section 12 is omitted entirely and replaced by an explicit notice in the relevant sections.

## Certification Output Scenarios

The pipeline emits different outputs depending on data completeness.

| Scenario | Condition | Output |
|---|---|---|
| **Full Certification with Statistics** | n ≥ 30 per fault category, Phase E ran | Complete 12-section report with statistical findings in Appendix A1–A5 |
| **Certification — Aggregation Only** | n < 30 per fault category | Complete 12-section report without Appendix; explicit "Statistical inadequacy" notice where hypothesis blocks would appear |
| **All Runs Failed** | Every run errored before producing usable telemetry | No certificate. Certifier not triggered. |
| **Runs Completed, No Metrics Extracted** | Runs completed but Phase 1 yielded no usable data | No certificate. Error output only. |

## The Pydantic Schema

The report's structure is enforced by Pydantic v2 models in `cert_builder/schema/certification_schema.py`.

```
CertificationReport
├── meta: Meta
│   ├── agent_name, agent_id, certification_date
│   ├── total_runs, categories
│   └── responsible_ai (RAI score, compliance status)
│
├── sections: list[Section]
│   └── Section
│       ├── number, id, title
│       ├── part (Agent Capability / Fault Injection / Limitations / Appendix)
│       └── content: list[ContentBlock]
│           └── ContentBlock (discriminated union on "type" field)
│               ├── TextBlock      { type: "text",       body, style? }
│               ├── HeadingBlock   { type: "heading",    title, detail? }
│               ├── TableBlock     { type: "table",      headers, rows, title? }
│               ├── CardBlock      { type: "card",       items, title? }
│               ├── ChartBlock     { type: "chart",      chart_type, data, axes }
│               ├── FindingsBlock  { type: "findings",   items[{severity, text}] }
│               └── AssessmentBlock{ type: "assessment", rating, confidence, explanation }
│
└── footer: str (minimal, just a non-breaking space)
```

The discriminated union on `type` means Pydantic validates each content block against the correct model. A `ChartBlock` must have `chart_type`, `data`, and `axes`. A `FindingsBlock` must have a list of items with valid severity values. If the report assembly produces something that doesn't match, validation catches it before the output is written.

## Extending the Report

If you need to add a new section or content type:

1. Add a new `ContentBlock` subclass to `certification_schema.py` with a unique `type` literal
2. Update the discriminated union in `ContentBlock`
3. Write a builder (computation or narrative) that produces that block type
4. Wire the builder into `report_assembler.py`
5. Update the cert reporter to render the new block type to HTML

The modular builder pattern means each section is largely independent. Adding a new section doesn't require touching existing builders.
