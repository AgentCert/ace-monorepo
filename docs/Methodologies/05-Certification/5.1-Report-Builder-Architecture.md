# 5.1 — Report Builder Architecture

Phase 3 (`cert_builder/`) is a parallel builder pipeline. It doesn't write a report top-to-bottom — instead, multiple independent builders each produce a piece of the report, and an assembler merges them at the end. This page covers how the builders work, what each one produces, and why the pipeline is structured this way.

---

## The Builder Pattern

Each builder is a Python class with a single responsibility: take a `ParsedContext` as input and return one or more `ContentBlock` objects (text, table, chart, assessment card, etc.) that slot into specific report sections.

Builders never call each other. They read from `ParsedContext` only. This makes them independently testable and safe to run in parallel.

---

## Phase 3 Sub-Phases

### 3.1 — Ingestion

Before any builder runs, the raw `CertificationScorecard` JSON from Phase 2 is parsed into a structured `ParsedContext`:

- Agent metadata: name, ID, certification date, total runs
- Per-category metrics: what data exists and what is null
- Confidence levels per metric (based on sample size)
- Availability flags: which sections can be populated vs. which need a "data unavailable" notice

`ParsedContext` is the single input that all builders receive.

### 3.2 — Computation Builders (no LLM)

Six builders run in parallel, all deterministic Python:

| Builder | Output | Where It Appears |
|---|---|---|
| `ScoreCard` | 7-dimension radar scores (0–1 each) | Section 1, Section 4 |
| `Table` | Detection/mitigation rate tables with CI whiskers | Section 4, Section 8 |
| `Assessment` | Resilience, Security, Performance rating cards | Section 6 |
| `Card` | Key-value metadata blocks (agent info, experiment params) | Section 2 |
| `HardcodedLoader` | Static YAML: methodology reference, glossary | Section 2, Section 12 |
| `Chart` | Radar chart, grouped bar charts, line charts | Section 1, Section 4 |

The Chart builder depends on the ScoreCard output for dimension normalization — so it runs after the other five, not concurrently with them.

**The 7 ScoreCard dimensions:**

| Dimension | Derived From |
|---|---|
| Detection Rate | SLA-aware detection score per category (weighted by run count) |
| Mitigation Rate | SLA-aware mitigation score per category (weighted by run count) |
| Action Correctness | Mean action correctness (0–1) |
| Reasoning Quality | Mean reasoning score, normalized 0–1 |
| Safety (RAI) | RAI compliance rate |
| Hallucination Ctrl | 1 − max hallucination score, normalized (inverted so higher is better) |
| Privacy & Security | Composite: security_compliance_rate × pii_clean_rate × adversarial_clean_rate |

### 3.3 — Narrative Builders (LLM)

Seven LLM-powered builders produce the text portions of the report. Six run concurrently via `asyncio.gather()`:

| Builder | What It Writes | LLM Input |
|---|---|---|
| `ScopeMethodology` | Executive context, test setup, ground truth approach | ParsedContext metadata + static YAML |
| `KeyFindings` | High-level findings with severity tags (concern/good/note) | All scorecard dimensions + per-category summaries |
| `QualitativeMetrics` | Reasoning quality, hallucination patterns, plan adherence interpretation | Qualitative metric aggregates |
| `FaultAnalysis` | Per-category narrative (one LLM Council call per category) | Per-category parsed data |
| `Limitations` | System constraints, scope boundaries, recurring failure modes | All per-run known_limitations from Phase 1/2 |
| `FairnessScore` | Cross-category RAI consistency evaluation | Per-category RAI and safety metrics |

The seventh builder runs sequentially after the six above complete:

| Builder | What It Writes | Why Sequential |
|---|---|---|
| `Recommendations` | Evidence-based improvement roadmap, prioritized by impact | Explicitly reads Limitations output as input |

Each narrative builder reports its own token usage. These are accumulated into `phase_3_tokens` and reported in Section 11.

### 3.4 — Assembly

The assembler merges all Phase 3.1 + 3.2 + 3.3 outputs into a single `CertificationReport`.

Each of the 12 sections is assembled from a combination of:
- Computed content (tables, charts, cards)
- LLM narratives (text blocks, findings blocks)
- Static content (methodology YAML, glossary)
- Optional hypothesis results (if Phase E ran — injected as Appendix A1–A5)

After assembly, **Pydantic v2 validation** runs on the complete output. If any section fails validation — a missing required field, an invalid severity value, a chart block without axes — the pipeline raises an error rather than writing a malformed report.

## Configuration

The builders read from `cert_builder/config/`:

| Config File | What It Controls |
|---|---|
| `narrative_config.yaml` | Model selection, temperature, max tokens for each narrative builder |
| `scorecard_config.json` | Dimension weights and which metrics map to which dimension |
| `section_config.yaml` | Which sections get which content types |

The LLM narrative builders all use the extraction model from `configs/configs.json` by default. You can override per-builder in `narrative_config.yaml` to route heavier analysis (e.g., FaultAnalysis) to the reasoning model.

---

## Adding a New Section or Builder

1. Add a new `ContentBlock` subclass to `cert_builder/schema/certification_schema.py` with a unique `type` literal
2. Add it to the discriminated union in `ContentBlock`
3. Write a builder class (computation or narrative) that produces the new block
4. Register the builder in `report_assembler.py`
5. Add HTML rendering for the new block type in the cert reporter

Each builder is independent. Adding one doesn't require touching existing builders.
