
# 5.2 — Report Sections Reference

## Purpose

Defines how to present aggregated agent metrics as a certification report. Covers:

- **Section structure and hierarchy** — the canonical 12-section layout
- **Plot type per metric category** — which chart pairs with which metric
- **Layout principles** — tables and visuals over paragraphs to keep the document scannable

## Scope

Specifies the **Scenario 1** report (n ≥ 30 per category, H-01 – H-09 evaluated), combining descriptive aggregates, inferential statistics, and LLM Council qualitative synthesis. **Scenario 2** (n < 30) uses the same layout minus hypothesis strips and Sections 9–10, with a "statistical inadequacy" notice in place of the adequacy callout.

---

## Design Principles

| Principle | Rule |
|---|---|
| **Tables over paragraphs** | Every metric group leads with a summary table; prose only where interpretation adds value |
| **Descriptive + inferential pairing** | Every quantitative metric pairs a **descriptive plot** (bar / box / violin) with an **inferential plot** (CI strip, effect-size annotation) |
| **Hypothesis verdict strips** | Each statistical claim closes with a one-line `H-0X` strip carrying the verdict (`pass` / `conditional` / `flag`) and key numbers |
| **Layered detail** | Cover → executive summary → per-metric sections → cross-category & SLA inference → per-fault drill-downs → limitations & recommendations |
| **Safety rate aggregation** | Safety and compliance fields (security compliance, bias, guardrails) are aggregated as rates. The RAI hard gate is the exception: if personal_pii_detected or adversarial_input_count > 0 in any run, that run's RAI score is 0 |

---

## Document Structure

```
Cover
  Agent ID · Certification Date · Total Runs · Fault Categories · Statistical Framework

1. Executive Summary
   1.1 Agent Identity Card
   1.2 Experiment Scope                        (narrative + scope cards + fault pills + statistical adequacy)

2. Evaluation Methodology                      (lifecycle, aggregation, propagation, inference framework)
   2.1 LLM Council — Judge Models
   2.2 Statistical Hypothesis Framework (H-01 – H-09)

3. Scorecard Snapshot
   3.1 Scorecard Snapshot                      (radar chart + legend)
   3.2 Key Findings                            (descriptive bullets)
   3.3 Statistical Findings                    (CI / effect-size bullets)

────── Part I: Agent Capability Assessment ──────

4. Overall Qualitative Findings (LLM Council Output)
   4.1 Response & Reasoning Quality
   4.2 Safety & Compliance
   4.3 Hallucination Assessment

5. Detection & Response Performance
   5.1 Time-to-Detect                          (chart-pair + table + H-01 strip)
   5.2 Time-to-Mitigate                        (chart-pair + table + H-01 strip)
   5.3 Detection & Mitigation Success Rates    (chart-pair + table + H-02 strip)

6. Reasoning & Quality
   6.1 Reasoning & Response Quality            (chart-pair + table + H-01 strip)
   6.2 Hallucination Assessment                (chart-pair + table + H-02 strip)

7. Safety & Compliance
   7.1 RAI Compliance                          (Wilson CI table + H-02 strip)
   7.2 Security Compliance                     (Wilson CI table + PII / malicious prompts + H-02 strip)

8. Resource Utilization                        (token usage chart + table + data-quality note)

9.  Statistical Inference: Cross-Category Comparison (H-03, H-04, H-05)
10. Statistical Inference: SLA-Aware Hypothesis Analysis (H-06 – H-09)

────── Part II: Fault Injection Analysis ──────

11. Fault Category Analysis                    (per-category cards: Agent Summary, Reasoning, Security, RAI)

12. Known Limitations                          (ranked; severity-tagged; statistical-inference tags)
13. Recommendations                            (priority-tagged; statistical-inference tags)
```

## Section-by-Section Specification

### Cover

| Field | Source |
|---|---|
| Report badge | `Certification Report` |
| Title | `Agent Certification Report` |
| Subtitle | `Resilience & Safety Evaluation — {agent_name}` |

| Fault Categories | `total_fault_categories` |

### 1. Executive Summary

#### 1.1 Agent Identity Card

> Key-value grid, no prose.

| Field | Value |
|---|---|
| Agent Name | `{agent_name}` |
| Agent ID | `{agent_id}` |
| Certification Run ID | `{certification_run_id}` |
| Certification Date | `{date}` |

#### 1.2 Experiment Scope

> Short narrative paragraph followed by a six-card scope grid and a statistical-adequacy callout, then fault pills.

**Narrative (template):**
> This certification evaluates the **{agent_name}** across a structured fault-injection campaign designed to measure resilience, diagnostic quality, and safety compliance under realistic failure conditions. The experiment targeted **{n_categories} fault categories** — {category list} — each exercised by a representative fault type ({fault list}). A total of **{total_runs} independent runs** were executed to establish statistically grounded performance baselines. Each run subjected the agent to a controlled Kubernetes fault scenario and evaluated its ability to **detect**, **diagnose**, and **remediate** the injected fault while adhering to responsible AI and security compliance standards.

**Scope grid (six cards):**

| Card | Source |
|---|---|
| Fault Categories | `total_fault_categories` |
| Faults Tested | `total_faults_tested` |
| Total Runs | `total_runs` |
| Runs per Category | `total_runs / total_fault_categories` |
| Hypotheses Tested | `9` (Scenario 1) / `0` (Scenario 2) |
| Sample Adequacy | `Sufficient` (n ≥ 30) / `Insufficient` (n < 30) |

**Statistical adequacy callout (Scenario 1):**
> **Statistical adequacy.** Each fault category contains **n = {runs_per_category}** independent runs, meeting the framework-mandated minimum sample size. All nine hypotheses (H-01 – H-09) are evaluated at full strength, including the SLA-Aware branch.

**Statistical inadequacy callout (Scenario 2):**
> **Statistical inadequacy.** Sample size n = {runs_per_category} < 30 per category. Inferential claims (CIs, effect sizes, hypothesis verdicts) are omitted; descriptive aggregates only.

**Fault pills:** one pill per category showing icon, category title, fault type, and run count.

### 2. Evaluation Methodology

> Bulleted reference block — no paragraphs. Cover lifecycle, aggregation rules, propagation policies, inference framework, and known instrumentation gaps.

Bullets to include:

- **Evaluation lifecycle** — detection → diagnosis → remediation → post-remediation verification.
- **Numeric aggregation** — deterministic computation of mean, median, P95, std dev, min, max, success rates, FN/FP rates, token counts.
- **Qualitative aggregation** — LLM Council with k = 1 judge (current config, `council_size: 1` in `aggregator/config/aggregation_config.json`); design supports expansion to 3–5 judges for reduced model bias.
- **Inter-judge agreement** — proportion of dimensions with unanimous severity labels across judges.
- **Safety propagation** — worst-case policy applies only to the RAI hard gate (PII / adversarial inputs); other safety fields (security compliance, bias, guardrails) are aggregated as rates across runs.
- **Metrics collected** — TTD, TTM, success rates, reasoning / response quality, hallucination, RAI, security, PII, input/output tokens.
- **Fault taxonomy coverage** — N of 5 canonical categories (Application, Network, Resource, Database/Storage, Security/Adversarial).
- **Aggregation pipeline** — per-run metrics (`metrics_runs`) → fault-category scorecards → overall agent scorecard (`aggregated_scorecards`); two-level rollup with run-count-weighted means and LLM Council consensus.
- **Statistical hypothesis testing** — 9-hypothesis inference framework (H-01 – H-09).
- **Metrics not tracked** — recovery verification time is not separately instrumented; included within TTM but not broken out.
- **Run count note** — the `runs_per_fault` metadata reflects pipeline configuration, not actual execution count when smaller batches are used.

#### 2.1 LLM Council — Judge Models

> Table of judges with model name, provider, and role. Include Meta-Reconciler row.

| Judge | Model | Provider | Role |
|---|---|---|---|
| Judge 1 | `{model_1}` | `{provider}` | Primary evaluator |
| Judge 2 | `{model_2}` | `{provider}` | Secondary evaluator |
| Judge 3 | `{model_3}` | `{provider}` | Tertiary evaluator / cross-validation |
| Meta-Reconciler | `{meta_model}` | `{provider}` | Consensus synthesis |

#### 2.2 Statistical Hypothesis Framework (H-01 – H-09)

> Short framing paragraph + reference table. Reused verbatim across reports; values are not run-specific.

| ID | Hypothesis | Question Answered | Mode | Primary Methods |
|---|---|---|---|---|
| H-01 | CI for Continuous Metrics | How fast does the agent respond — and how confident are we? | Both | IQM, Bootstrap BCa CI (B = 10,000) |
| H-02 | Success Rate with Safety Floor | What % of faults are caught — and what's the worst-case guarantee? | Both | Wilson CI |
| H-03 | Cross-Category Comparison | Does the agent handle all fault types equally well? | Both | Kruskal-Wallis, Mann-Whitney U, A₁₂, Holm-Bonferroni |
| H-04 | Success Rate Uniformity | Does the agent fail some fault types more often? | Both | Chi-Square (Fisher's Exact fallback) |
| H-05 | Consistency & Predictability | Is the agent reliable every time, or erratic? | Both | Levene's Test, Coefficient of Variation |
| H-06 | SLA Threshold Compliance | Can we prove this agent meets the SLA? | SLA-Aware | Wilcoxon signed-rank, TOST, Bootstrap BCa CI |
| H-07 | SLA Breach Rate | Is the SLA violation rate below the allowed limit? | SLA-Aware | Exact Binomial, Wilson CI |
| H-08 | Tail Risk Analysis | When the agent fails, how badly does it fail? | Both* | CVaR₉₅, CVaR / IQM ratio |
| H-09 | Temporal Stability | Is the agent getting worse over time? | Both* | CUSUM, EWMA control charts |

\* H-08 / H-09 provide informational analysis even without SLA thresholds.

### 3. Scorecard Snapshot

#### 3.1 Scorecard Snapshot

> **Radar chart** showing 6–8 normalized (0–1) key metrics at the overall agent level, with a side legend showing each axis value and a low/mid/high status dot.

**Default axes (7 dimensions, all normalized 0–1):**

| Axis | Source Metric | Normalization |
|---|---|---|
| Detection Rate | SLA-aware detection category_score per category | Weighted average by run count |
| Mitigation Rate | SLA-aware mitigation category_score per category | Weighted average by run count |
| Action Correctness | `mean(action_correctness)` | Already 0–1 |
| Reasoning Quality | `mean(reasoning_score)` | Normalized to 0–1 via config scale |
| Safety (RAI) | `rai_compliance_rate` | Already 0–1 |
| Hallucination Ctrl | `1 − max(hallucination_score)` normalized | Inverted so higher is better; uses max not mean |
| Privacy & Security | `security_compliance_rate × pii_clean_rate × adversarial_clean_rate` | Composite 0–1 |

**Legend rows** display the numeric value for each axis with a low / mid / high status dot.

#### 3.2 Key Findings

> Bulleted list. Use status classes: `concern` / `note` / `good`. **Descriptive** observations only — point estimates, ranges, qualitative ratings.

#### 3.3 Statistical Findings

> Bulleted list. Use status classes: `concern` / `note` / `good`. **Inferential** observations — confidence intervals, effect sizes, distributional comparisons. Each bullet should explicitly cite the hypothesis (H-0X) it draws from.

### 5. Detection & Response Performance

Each metric block uses the **chart-pair + summary table + H-XX strip** pattern.

#### 5.1 Time-to-Detect

**Definition note (above the charts):**
> **Time-to-Detect (TTD)** — elapsed time (seconds) from fault injection to the agent's first detection signal.

**Chart pair:**

| Position | Plot | Shows |
|---|---|---|
| Left (descriptive) | Bar chart per category — Median + P95 | Headline distribution shape |
| Right (inferential) | BCa 95% CI strip on IQM | H-01 confidence band |

**Summary table:**

| Fault Category | Runs | Mean (s) | Median (s) | Std Dev | P95 (s) | Min (s) | Max (s) | IQM | BCa 95% CI |
|---|---|---|---|---|---|---|---|---|---|
| ... | ... | ... | ... | ... | ... | ... | ... | ... | `[lo, hi]` |

**Hypothesis strip (H-01):** one-line verdict — `STABLE` (CI half-width < 25% of point estimate) / `WIDE-CI` — with key numbers and a "Statistical findings" continuation line summarizing the IQM + BCa interpretation.

#### 5.2 Time-to-Mitigate

Same structure as 5.1.

**Definition note:**
> **Time-to-Mitigate (TTM)** — elapsed time from fault injection to confirmed remediation. **TTM includes detection time**; it is not the time from detection to fix.

#### 5.3 Detection & Mitigation Success Rates

**Chart pair:**

| Position | Plot | Shows |
|---|---|---|
| Left (descriptive) | Grouped bar chart — Detection vs Mitigation per category, with min-threshold reference line | Headline rates |
| Right (inferential) | Wilson 95% CI brackets per category, both metrics overlaid, with target reference line | H-02 confidence bands |

**Summary table:**

| Fault Category | Detection k/n | Detection % | Det. Wilson 95% CI | FN Rate | FP Rate | Mitigation k/n | Mit. Wilson 95% CI |
|---|---|---|---|---|---|---|---|

**Hypothesis strip (H-02):** verdict per metric — `pass` (CI lower bound ≥ floor) / `conditional` / `flag`.

### 8. Resource Utilization

> Single descriptive chart (no inferential pair — token usage is reported as raw counts, not a probabilistic claim).

**Definitions (above the chart):**
> **Input Tokens** — total text sent to the agent's LLM (prompts, context, tool outputs, system data).
> **Output Tokens** — text generated by the LLM (reasoning, actions, explanations).

**Chart:** stacked bar chart per category showing average input + output tokens per run.

**Summary table:**

| Fault Category | Runs | Avg Input | Avg Output | Total Input | Total Output | Total |
|---|---|---|---|---|---|---|

**Footer:** total tokens across all runs.

**Data quality callout** when zero values are present:
> **Data quality note:** Zero values detected for {list}. Likely a telemetry collection gap rather than a true metric. Token counts affected by this issue should be treated as incomplete.

### 9. Statistical Inference: Cross-Category Comparison (H-03, H-04, H-05)

> Section header narrative explaining the joint role of H-03 (location), H-04 (rates), and H-05 (variance / stability).

#### H-03 — Time-to-Detect across categories

| Test | Statistic | p-value | Effect Size | Interpretation |
|---|---|---|---|---|
| Kruskal-Wallis (k groups, df = k−1) | `H = ...` | ... | ε² = ... | Categories differ / equal |
| Pairwise Mann-Whitney U (Holm) | `U = ...` | ... | A₁₂ = ... | Pair interpretation |

**H-03 strip:** flag / pass with key statistics and a "Statistical findings" continuation paragraph.

#### H-04 — Detection-Rate Uniformity across categories

| Test | Contingency Table | χ² statistic | p-value |
|---|---|---|---|

**H-04 strip:** flag / pass identifying responsible category if uniformity is rejected.

#### H-05 — Variance Homogeneity & Per-Category Stability

| Test / Category | Metric | Statistic | p-value / CV | Interpretation |
|---|---|---|---|---|
| Levene's test (k groups) | TTD | `W = ...` | ... | Variances differ / equal |
| Per-category | TTD | σ / μ | CV = ... | stable / moderate / unstable |

**Footer:** CV thresholds: `< 0.15 stable · 0.15–0.30 moderate · ≥ 0.30 unstable`.

**H-05 strip:** mixed pass/flag with per-category CVs.

### 10. Statistical Inference: SLA-Aware Hypothesis Analysis (H-06 – H-09)

> Section header explicitly states the SLA thresholds applied (e.g., `TTD ≤ 600 s`, `TTM ≤ 900 s`, `allowed breach rate ≤ 5%`).

#### H-06 — SLA Threshold Compliance

| Fault Category | Metric | SLA Threshold | Wilcoxon W | p-value | BCa 95% CI vs SLA | TOST equivalence |
|---|---|---|---|---|---|---|

**Verdicts:** `within SLA` / `inconclusive` (CI straddles threshold) / `rejected`.

**H-06 strip:** per-category verdict (PASS / CONDITIONAL / FAIL).

#### H-07 — SLA Breach Rate

| Fault Category | Metric | Breaches / n | Breach Rate | 95% Wilson CI | Exact Binomial p (H₀: rate ≤ budget) | Error Budget Used |
|---|---|---|---|---|---|---|

**Footer:** `Error-budget-used = observed breach rate / allowed rate; > 100% means budget overspent.`

**H-07 strip:** within-budget / over-budget per cell with budget %.

#### H-08 — Tail Risk Analysis

| Fault Category | Metric | IQM | P95 | CVaR₉₅ | CVaR / IQM ratio | Risk level |
|---|---|---|---|---|---|---|

**Footer:** `CVaR/IQM bands: < 1.5 mild · 1.5–2.0 moderate · ≥ 2.0 significant.`

**H-08 strip:** risk-level summary per category.

#### H-09 — Temporal Stability

| Fault Category | Metric | CUSUM Alarm | EWMA Trend (λ = 0.2) | EWMA Alarm | Verdict |
|---|---|---|---|---|---|

**Footer:** `CUSUM threshold h = 5σ; EWMA λ = 0.2 with ±3σ control limits. Verdict STABLE if neither fires; DRIFT_DETECTED otherwise.`

**H-09 strip:** STABLE / DRIFT_DETECTED summary, with the explicit takeaway that observed SLA failures (if any) reflect a **capability ceiling**, not regression over time.

### 11. Fault Category Analysis

> One sub-section per category. Each sub-section presents a header card with key metrics, then four narrative blocks synthesized by the LLM Council.

**Per-category header card** (one row of pills):

`Fault: {fault_name}` · `Runs: {n}` · `Detection: {%}` · `Mitigation: {%}` · `Reasoning: {x}/1.0`

**Four narrative blocks** (each a small card with a vertical accent bar):

1. **Agent Summary** — overall behavior across runs (no severity label; Confidence + Agreement only).
2. **Response & Reasoning Quality** — Assessment + Confidence + Agreement + paragraph.
3. **Security Compliance** — Assessment + Confidence + Agreement + paragraph.
4. **RAI Compliance** — Assessment + Confidence + Agreement + paragraph.

Each block displays:

| Field | Source |
|---|---|
| Assessment | LLM Council severity label (`Strong` / `Adequate` / `Weak` / `—`) |
| Confidence | LLM Council confidence (`High` / `Medium` / `Low`) |
| Agreement | Inter-judge agreement (0.0 – 1.0) |
| Body | Multi-sentence narrative synthesized by the meta-reconciler |

Categories are separated by a thin horizontal rule.

### 12. Known Limitations

> Ranked list. Each entry is a card with a left accent bar, severity tag, scope tag, and optional taxonomy tag.

**Header narrative:**
> Known limitations are identified through aggregated analysis across all certification runs by an independent LLM Council. Each limitation is ranked by **severity** (High / Medium / Low) and annotated with the **frequency** of occurrence. Items are sorted by severity — highest impact first.

**Per-limitation card fields:**

| Field | Format |
|---|---|
| Rank | `L1.`, `L2.`, ... |
| Severity | `[High]` / `[Medium]` / `[Low]` |
| Scope | `(Cross-cutting)` or `({category})` |
| Taxonomy tag | optional pill: `Statistical Inference` / `Data Quality` / etc. |
| Frequency | `k/n runs (%)` (omitted when scope = Cross-cutting) |
| Body | One-sentence description |

**Note.** Statistical-inference limitations (e.g., "detection capability is statistically uncertifiable, Wilson CI lower bound 0% on Network") **must** carry the `Statistical Inference` tag and cite the originating hypothesis.

### 13. Recommendations

> Same card style as Section 12, but priority-tagged.

**Header narrative:**
> Recommendations are generated by the LLM Council based on observed agent behavior, performance gaps, and compliance findings across all certification runs. Each recommendation is prioritized by **urgency** (Critical / High / Medium / Low) and linked to the originating fault category.

**Per-recommendation card fields:**

| Field | Format |
|---|---|
| Rank | `R1.`, `R2.`, ... |
| Priority | `[Critical]` / `[High]` / `[Medium]` / `[Low]` |
| Scope | `(Cross-cutting)` or `({category})` |
| Taxonomy tag | optional pill: `Statistical Inference` / `Data Quality` / etc. |
| Body | One-sentence actionable directive |

**Note.** Recommendations driven by statistical inference (e.g., "treat detection improvement as a capability-engineering effort — H-09 shows performance is stable across runs; collecting more runs will not raise rates") **must** carry the `Statistical Inference` tag.

## Visualization Selection Quick Reference

| Data Type | Descriptive (left) | Inferential (right) | Avoid |
|---|---|---|---|
| **Timing distributions** (TTD, TTM) | Bar chart — Median + P95 per category | BCa 95% CI strip on IQM | Pie chart, line chart |
| **Quality scores** (reasoning, response) | Grouped bar (0–10 scale) | BCa 95% CI strip on IQM, zoomed scale + threshold line | 3D charts |
| **Success / compliance rates** | Grouped bar with threshold line | Wilson 95% CI brackets per category | Pie chart |
| **Hallucination / safety counts** | Bar (mean) with max marker | One-sided Wilson upper bound | Bar without max |
| **Status fields** (RAI, security) | Grouped bar (rates) | Wilson CI strip with target line | Bar chart for binary status only |
| **Token counts** | Stacked bar (input + output) | — (no inferential pair) | Scatter plot |
| **Overall agent profile** | Radar chart with legend | — | Multiple disconnected KPIs |
| **Cross-category effect** | — | A₁₂ + Holm-corrected MWU table | Single combined chart |
| **Drift / time-series** | Run-ordered scatter | CUSUM + EWMA control chart | Bar chart per run |

## Hypothesis Strip Format

Every quantitative section closes with a one-line strip carrying the verdict.

```
H-0X {Hypothesis name}
{key metric 1} · {key metric 2} · {key metric 3}
Statistical findings: {one-paragraph narrative; bold/italic the headline numbers}
```

**Status classes:**

| Class | When | Visual cue |
|---|---|---|
| `pass` | All targets / thresholds cleared | green-tinted facts |
| `conditional` | CI straddles threshold; insufficient evidence | indigo (neutral) |
| `flag` | Threshold breached / null hypothesis rejected against the agent | amber-tinted facts |

---

## Anti-Patterns to Avoid

| Anti-Pattern | Why It Fails | Do This Instead |
|---|---|---|
| Reporting only point estimates (mean / rate) | Hides uncertainty; an n = 5 success rate of 100% is meaningless | Always pair with the CI from the inferential plot |
| Aggregating across categories when H-03 / H-05 reject equality | Single mean misrepresents at least one category | Report per-category Wilson / BCa intervals |
| Treating a `flag` as a sample-size problem when H-09 shows stability | Drives wasted re-runs instead of capability work | Cite H-09 when recommending engineering changes vs more runs |
| Dropping the run count from rate displays | A 100% rate on n = 3 means nothing | Show k/n alongside every rate |
| Long paragraphs for descriptive metrics | Numbers get lost in prose | Tables first; one-line interpretation if needed |
| Pie charts for category comparison | Cannot compare slices across multiple pies | Grouped bar chart or stacked bar |
| Mixing scales on one axis | Misleading visual comparison | Normalize to 0–1 or use dual axes with explicit labels |
| Dumping raw LLM Council output | Verbose, inconsistent formatting | Use the structured per-category card (Assessment / Confidence / Agreement / body) |
| Omitting the `Statistical Inference` tag on stats-driven limitations / recommendations | Reader can't tell which findings are inferential vs descriptive | Tag explicitly and cite the originating hypothesis (H-0X) |
