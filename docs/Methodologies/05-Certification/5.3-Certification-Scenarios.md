# 5.3 — Certification Decision Guide

When you run the AgentCert pipeline, which output you get depends on how much data you have and whether Phase E (hypothesis testing) ran. This page explains what each outcome looks like and what you should do about it.

For the full scenario definitions and conditions, see [2.6 Certification Scenarios](02-Experiment-Design/2.6-Certification-Scenarios).

---

## The Four Scenarios at a Glance

| Scenario | What Triggered It | What You Get |
|---|---|---|
| **S1 — Full Certification** | ≥ 30 runs per fault category + Phase E ran | 12-section report with statistical appendix (A1–A5) |
| **S2 — Certification, No Stats** | 1–29 runs per fault category | 12-section report with explicit "statistical inadequacy" notice where hypothesis blocks would appear |
| **S3 — No Certificate** | Every run failed before producing usable telemetry | No report. Upstream failure. Check your trace pipeline. |
| **S4 — Cert with Warnings** | Runs completed but Phase 1 extracted no usable metrics | Report exists but key sections show "data unavailable." Something went wrong in extraction. |

---

## What to Do in Each Case

### S1 — Full Certification

This is the definitive output. The report includes confidence intervals, hypothesis test results, and a certified lower-bound floor for detection and mitigation rates. This is the version suitable for handing to a customer or submitting for sign-off.

Check the statistical appendix (Section 12) to confirm hypothesis tests passed and effect sizes are meaningful, not just statistically significant.

### S2 — Certification, Aggregation Only

You have a real report — mean/median performance, detection rates, qualitative narratives — but the statistics are directional, not certified. The right response depends on your customer's requirements:

- If they need certified lower bounds, run more experiments to reach n ≥ 30 and re-run Phase E
- If the directional data is sufficient for their purposes, S2 is acceptable for a preliminary review

The "statistical inadequacy" notices in the report are explicit — the reader knows what's missing without any risk of misinterpretation.

### S3 — All Runs Failed

Nothing reached Phase 1. This is an infrastructure or trace collection problem, not an agent problem. Check:
- Was Langfuse receiving telemetry during the experiment?
- Did the trace file get generated at all?
- Are the trace spans tagged with fault context?

Don't re-run certification until you've confirmed the trace pipeline is healthy. See [2.5 Trace Collection and Preprocessing](02-Experiment-Design/2.5-Trace-Collection-and-Preprocessing) for what a valid trace looks like.

### S4 — Runs Completed, No Metrics Extracted

Traces exist but the metrics extractor (Phase 1) produced nothing usable. Common causes:
- Trace spans are missing the expected span types (GENERATION, TOOL_CALL)
- Fault injection timestamps are absent from the trace, so TTD/TTM can't be computed
- The fault bucket is empty — Phase 0 couldn't separate the interleaved events

A partial report may be emitted with "data unavailable" in the affected sections. Review the Phase 0 bucketing manifest to see what was classified and what wasn't.

---

## Choosing When to Stop Running Experiments

If you're midway through a campaign and asking "do I have enough runs?":

- **< 5 runs**: Results are directional only. Don't draw conclusions — you're in INSUFFICIENT confidence tier.
- **5–19 runs**: MEDIUM confidence. Useful for internal assessment; not ready for formal certification.
- **≥ 20 runs**: HIGH confidence. Certification-grade for S2 scenarios.
- **≥ 30 runs**: Required for Phase E and the S1 full statistical certification.

Phase E (hypothesis testing) requires at least 30 runs per fault category. If you have 30 runs for some categories but not others, Phase E will run for the categories that qualify and be skipped for the rest.
